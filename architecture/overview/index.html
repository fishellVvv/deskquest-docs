    Vista general de arquitectura¶

    El proyecto se puede entender como 4 capas que cooperan entre si.

    1. Capa de producto¶

    La experiencia visible se divide en tres niveles:

    • idle
    • dashboard
    • game

    Hoy las tres capas ya existen.

    • idle vive en escritorio
    • dashboard centraliza gestion y lectura del reino
    • game ya tiene un primer slice jugable real: Dirigir tala v1

    2. Capa frontend¶

    Vive en src/ y usa React + TypeScript.

    Sus responsabilidades son:

    • renderizar ventanas
    • aplicar el tema resuelto de la dimension activa
    • mostrar configuracion y telemetria
    • coordinar idle y dashboard

    Punto de entrada¶

    • src/main.tsx
    • src/app/App.tsx

    Providers principales¶

    • ConfigurationProvider preferencias persistentes del usuario
    • ThemeProvider dimension activa + dimensiones instaladas + fallback
    • CoreI18nProvider textos base del sistema
    • TelemetryProvider pulsaciones y reset
    • WorldProvider fuente de verdad del reino temprano, persistencia y sincronizacion entre ventanas

    3. Capa de dominio¶

    Vive en src/domain/ y representa el estado neutral del mundo.

    La idea es que aqui no viva la ambientacion medieval o lunar, sino conceptos reutilizables:

    • configuracion
    • telemetria
    • estado del mundo
    • balance
    • progreso de actividades
    • misiones manuales tempranas como Dirigir tala

    4. Capa nativa¶

    Vive en src-tauri/ y convierte el proyecto en una app de escritorio real.

    Sus responsabilidades son:

    • abrir ventanas
    • posicionar el idle
    • permitir importacion e instalacion de dimensiones
    • leer assets del idle
    • exponer comandos al frontend

    Las piezas nativas mas importantes hoy son:

    • src-tauri/src/lib.rs
    • src-tauri/src/dimensions.rs

    Relacion entre capas¶

    El flujo normal hoy es este:

    1. Tauri arranca la app.
    2. React monta App.
    3. ConfigurationProvider carga preferencias.
    4. ThemeProvider resuelve la dimension activa y el tema visual.
    5. resolve-app-view decide si la ventana actual es idle o dashboard.
    6. idle usa la configuracion de posicion/tamano y pinta la base.
    7. Al pulsar la puerta, se abre el dashboard.
    8. Ambos comparten configuracion, dimension activa y telemetria.

    Nota importante sobre el idle raster¶

    Cuando el idle usa arte PNG, el tamano que realmente importa no es solo el de la ventana Tauri, sino el viewport CSS final del WebView. Esa calibracion ya esta documentada en:

    • Calibracion del idle raster

    Sistema de dimensiones¶

    Este es el sistema mas maduro del proyecto ahora mismo.

    Regla actual¶

    • Medieval es la unica dimension integrada
    • las demas se importan como .dimension
    • al importarlas se instalan descomprimidas en una carpeta interna
    • desde la configuracion se pueden importar y activar una a una

    Fallback¶

    Si una dimension falla o es incompleta:

    • la app cae a builtin:medieval

    Principio clave¶

    La logica del mundo y la ambientacion no deben mezclarse.

    Identidad de aplicacion¶

    La identidad aprobada del producto y del runtime ya no depende del placeholder historico test_game. La fuente de verdad para nombre visible, slug tecnico y app id objetivo queda fijada en:

    • Identidad de aplicacion
    • Plan de migracion a DeskQuest

    Eso significa:

    • el dominio debe seguir siendo neutral
    • la dimension decide colores, nombres, textos y assets
    • el sistema siempre debe poder volver a Medieval
