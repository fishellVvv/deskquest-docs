    Preparacion de M4b - cleanup residual de legacy¶

    Estado¶

    • estado: cerrado
    • milestone asociado: M4b
    • abre milestone: si

    Objetivo¶

    Cerrar los flecos de legacy que ya no pertenecen al nucleo de M4, pero que siguen ensuciando la base tecnica despues del cierre del milestone.

    M4b no reabre el frente de world persistence + sync.

    Su funcion es rematar el legacy residual que ya no protege nada vivo del producto ni del refactor principal.

    Alcance¶

    M4b queda limitado a cuatro cortes cortos y deliberados:

    1. retirar la migracion nativa del perfil testgame
    2. retirar el puente legacy de settlementName
    3. simplificar browser-storage-migration.ts
    4. decidir si se retira el mapa legacy de ids de dimension en runtime

    No entra aqui:

    • un nuevo refactor horizontal de stores browser
    • unificacion grande de infraestructura
    • cambios de UI
    • trabajo de producto o midgame

    Lectura de partida¶

    Tras cerrar M4, el repo ya no mantiene compatibilidad general con test_game.* en los stores activos del runtime.

    Sin embargo, quedan aun cuatro flecos historicos que no encajan bien ni como compatibilidad de producto ni como legacy transicional del refactor:

    1. Migracion nativa del perfil testgame¶

    Hoy sigue viva en:

    • app_identity.rs
    • lib.rs

    Lectura:

    • si test_game no llego a distribuirse fuera del desarrollo, esta migracion ya no protege una superficie real de producto
    • hoy actua mas como complejidad de arranque que como soporte necesario

    2. Puente legacy de settlementName¶

    Hoy sigue vivo en:

    • configuration-store.ts
    • world-legacy-settlement-name.ts
    • WorldProvider.tsx

    Lectura:

    • el nombre del asentamiento ya pertenece al world
    • el antiguo settlementName de configuracion ya no parece producir valor real
    • mantener ese puente sigue cargando una migracion que ya no aporta al comportamiento normal del juego

    3. Helper residual de migracion browser¶

    Hoy sigue vivo en:

    • browser-storage-migration.ts

    Lectura:

    • parte del helper ya no tiene consumidores vivos
    • la utilidad que sigue teniendo valor puede probablemente quedar reducida a una superficie mucho mas pequena

    4. Mapa legacy de ids de dimension¶

    Hoy sigue vivo en:

    • runtime-theme-registry.ts

    Lectura:

    • sigue remapeando ids antiguos como medieval, lunar-base o builtin:lunar-base
    • es el unico punto del bloque con una duda funcional residual, porque si se retira deja de aceptarse estado persistido antiguo de seleccion de dimension

    Orden recomendado¶

    1. M4b-1 - retirar la migracion nativa de perfil testgame
    2. M4b-2 - retirar el puente legacy de settlementName
    3. M4b-3 - simplificar browser-storage-migration.ts
    4. M4b-4 - decidir y, si procede, retirar el mapa legacy de ids de dimension

    Criterio de cierre¶

    M4b solo deberia darse por cerrado cuando:

    1. los cuatro puntos residuales hayan sido revisados explicitamente
    2. toda compatibilidad que se mantenga siga teniendo justificacion escrita
    3. npm run build siga limpio tras los cortes que entren
    4. el smoke test final no muestre regresiones nuevas
    5. la documentacion activa deje claro que legacy sigue vivo y cual ya no

    Relacion con la politica de legacy¶

    M4b no cambia la politica ya fijada en:

    • Politica de legacy para M4

    La aplica a un bloque residual posterior al cierre de M4.

    Siguiente paso recomendado¶

    Abrir M4b con el corte mas claro y menos ambiguo del lote:

    1. revisar app_identity.rs
    2. retirar la migracion nativa de perfil testgame si no protege ningun dato vivo real
    3. sincronizar en el mismo bloque la documentacion operativa de instalacion y datos de usuario

    Bloque 1 - retirada de la migracion nativa de perfil testgame (completado)¶

    El primer corte real de M4b ya queda ejecutado.

    Se completa asi:

    • app_identity.rs sale del runtime nativo
    • lib.rs deja de invocar una migracion automatica del perfil io.github.fishellvvv.testgame
    • la documentacion operativa de instalacion y datos de usuario deja de presentar esa migracion como comportamiento vivo
    • las paginas de identidad pasan a tratarla como una regla historica del bloque de rename, no como compatibilidad activa del producto

    Validacion:

    • cargo check limpio
    • npm run build limpio

    Implicacion:

    • el perfil io.github.fishellvvv.testgame deja de existir como ruta que el runtime intente absorber automaticamente
    • pasa a tratarse ya solo como legado historico, soporte manual o backup
    • el siguiente corte natural de M4b pasa a ser el puente legacy de settlementName

    Bloque 2 - retirada del puente legacy de settlementName (completado)¶

    El segundo corte real de M4b ya queda ejecutado.

    Se completa asi:

    • configuration-store.ts deja de conservar settlementName como preferencia persistente
    • world-legacy-settlement-name.ts sale del codigo vivo
    • WorldProvider.tsx deja de coordinar la migracion transicional desde configuracion hacia world

    Validacion:

    • npm run build limpio

    Implicacion:

    • el nombre del asentamiento queda ya modelado solo como parte del world
    • configuration deja de arrastrar un campo que ya no representa una preferencia real del usuario
    • el siguiente corte natural de M4b pasa a ser la simplificacion de browser-storage-migration.ts

    Bloque 3 - simplificacion de browser-storage-migration.ts (completado)¶

    El tercer corte real de M4b ya queda ejecutado.

    Se completa asi:

    • browser-storage-migration.ts deja de arrastrar helpers de carga y limpieza legacy sin consumidores vivos
    • el modulo queda reducido a storageEventMatchesAnyKey, que sigue siendo la unica utilidad compartida usada por los stores browser activos

    Validacion:

    • npm run build limpio

    Implicacion:

    • el helper deja ya de presentarse como una capa de migracion browser general cuando en realidad solo estaba sosteniendo una comprobacion comun de eventos
    • el siguiente y ultimo fleco real de M4b pasa a ser decidir si se retira el mapa legacy de ids de dimension en runtime

    Bloque 4 - retirada del mapa legacy de ids de dimension (completado)¶

    El cuarto corte real de M4b ya queda ejecutado.

    Se completa asi:

    • runtime-theme-registry.ts deja de remapear ids heredados como medieval, lunar-base y builtin:lunar-base
    • active-dimension-storage.ts deja de normalizar ids persistidos antiguos y pasa a leer solo el runtime id guardado tal cual
    • themes/index.ts y typescript-api.ts dejan de exponer normalizeRuntimeThemeId como API publica

    Validacion:

    • npm run build limpio

    Implicacion:

    • el runtime deja de aceptar ids de dimension anteriores a 0.1.0-beta.2
    • M4b ya no tiene flecos legacy pendientes dentro del repo
    • el siguiente paso natural pasa a ser la validacion final del milestone y su cierre documental

    Cierre¶

    M4b queda ya cerrado.

    Motivo:

    • los cuatro flecos residuales del milestone han sido revisados y resueltos
    • npm run build sigue limpio tras el ultimo corte
    • el smoke test final ya ha pasado sin regresiones visibles
    • no quedan compatibilidades pre-beta.2 vivas dentro del repo cuya permanencia no tenga justificacion escrita

    Resultado:

    • el bloque residual de legacy queda rematado
    • el repo ya no mantiene compatibilidad general con identidades o formatos anteriores a 0.1.0-beta.2
    • el siguiente paso ya no pertenece a M4b, sino a decidir que bloque corto conviene abrir antes de entrar en M5
