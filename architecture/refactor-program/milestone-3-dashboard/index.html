    Milestone 3 - Dashboard surfaces¶

    Estado¶

    • estado: cerrado
    • depende de: M2 ya cerrado
    • foco: DashboardCourtyard y DashboardSettingsRoom
    • situacion actual: DashboardSettingsRoom ya delega integraciones pesadas y el patio ya externaliza seleccion local, wiring de Dirigir tala, overview ledger, ledger del edificio principal, ledger del edificio seleccionado, hoja de nombrado inicial y overlay de QA; ademas, el dashboard ya no enruta sus lecturas de world por la fachada world-state; el smoke test final del milestone ya ha pasado sin regresiones visibles

    Objetivo¶

    Partir las superficies grandes del dashboard para que dejen de concentrar demasiada coordinacion visual, wiring y pequenas reglas operativas en el mismo archivo.

    M3 no busca rehacer el dashboard ni redisenar sus flujos. Su objetivo es:

    • convertir DashboardCourtyard.tsx en una shell de composicion mas clara
    • convertir DashboardSettingsRoom.tsx en una superficie de settings mas modular
    • extraer integraciones y coordinacion donde ya no pertenezcan al componente visual
    • preparar el terreno para que M4 no tenga que convivir con hotspots de UI que sigan creciendo

    Hotspots actuales¶

    DashboardCourtyard.tsx¶

    Hoy mezcla al mismo tiempo:

    • shell visual del patio y el pergamino
    • seleccion de plots y alternancia con settings
    • flujo de nombrado inicial del asentamiento
    • herramientas QA y shortcuts ocultos
    • wiring completo del slice Dirigir tala
    • ledger del overview, del edificio seleccionado y del edificio principal

    No es solo un archivo grande: es una superficie que hoy contiene varias subexperiencias del dashboard.

    DashboardSettingsRoom.tsx¶

    Hoy mezcla:

    • shell de configuracion en modo full y embedded
    • catalogo de secciones y controles
    • integracion Tauri para monitores
    • importacion, activacion y eliminacion de dimensiones
    • apertura de documentacion publica
    • acciones destructivas de Danger Zone

    Su forma es mas modular que la del patio, pero sigue concentrando demasiadas integraciones en la misma superficie.

    DashboardWindowView.tsx¶

    Hoy no es un hotspot prioritario.

    Su papel actual como shell de ventana es suficientemente claro, asi que M3 no deberia repartirlo sin necesidad.

    Que si entra en M3¶

    • extraer integraciones y acciones de DashboardSettingsRoom
    • extraer el estado de seleccion del patio y su wiring de navegacion local
    • extraer la coordinacion del slice Dirigir tala fuera del patio
    • partir los distintos ledgers del patio en piezas de composicion mas pequenas
    • reducir dependencias de dashboard contra la fachada temporal world-state.ts

    Que no entra en M3¶

    • rehacer WorldProvider
    • abrir platform/world/
    • rehacer persistencia o sincronizacion
    • redisenar Dirigir tala
    • introducir sistemas nuevos de producto
    • tocar DashboardWindowView.tsx salvo wiring minimo si una extraccion lo exige

    Orden recomendado¶

    1. descargar primero DashboardSettingsRoom por integraciones, no por JSX
    2. extraer despues la seleccion local del patio
    3. extraer luego la coordinacion del slice Dirigir tala
    4. partir por ultimo los distintos ledgers y overlays del patio
    5. cerrar el milestone con limpieza de imports y validacion final

    Primer corte recomendado¶

    El primer bloque real de M3 deberia ser:

    1. abrir rutas auxiliares para DashboardSettingsRoom
    2. sacar de ahi:
    3. carga de monitores del idle
    4. acciones de documentacion y Danger Zone
    5. acciones de importacion, activacion y eliminacion de dimensiones
    6. dejar DashboardSettingsRoom.tsx como composicion de secciones y controles
    7. validar npm run build

    Bloque 1 ya ejecutado¶

    • DashboardSettingsRoom.tsx deja de contener la mayor parte de sus integraciones y handlers pesados
    • nacen:
    • useDashboardIdleMonitorOptions.ts
    • useDashboardDocumentationActions.ts
    • useDashboardDimensionActions.ts
    • useDashboardDangerZoneActions.ts
    • el componente principal ya se parece mas a una composicion de secciones que a una superficie que mezcla shell, Tauri, dimensiones, documentacion y acciones destructivas
    • npm run build vuelve a salir limpio tras este primer corte real de M3

    Bloque 2 ya ejecutado¶

    • nace useCourtyardSelection.ts para encapsular la seleccion local del patio, la memoria del ultimo plot no-settings y el reseteo de seleccion por fase
    • DashboardCourtyard.tsx deja de gobernar directamente esa memoria local y el retorno al plot por defecto
    • el patio sigue siendo un hotspot grande, pero ya pierde una pieza de coordinacion de navegacion que no necesitaba vivir en el componente principal
    • npm run build vuelve a salir limpio tras este segundo corte real de M3

    Bloque 3 ya ejecutado¶

    • nace useCourtyardLoggingMission.ts para encapsular setup, mission state, timer, atajos, settle, avance de nivel y coordinacion del slice Dirigir tala
    • DashboardCourtyard.tsx deja de gobernar directamente el wiring de mission y pasa a consumir ese flujo desde un controller local de features/dashboard
    • el patio sigue siendo un hotspot grande, pero ya pierde otra responsabilidad transversal que mezclaba UI, atajos, estado transitorio y progreso del slice activo en el mismo archivo
    • npm run build vuelve a salir limpio tras este tercer corte real de M3

    Bloque 4 ya ejecutado¶

    • nace CourtyardOverviewLedger.tsx para encapsular el overview del patio, sus recursos, acciones disponibles y el footer de progreso de prestigio
    • DashboardCourtyard.tsx deja de renderizar directamente ese ledger y gana un corte de composicion mucho mas claro en su rama principal
    • el overview queda ya separado sin cambiar comportamiento ni clases visuales, lo que permite seguir troceando el patio por superficies del pergamino y no por helpers sueltos
    • npm run build vuelve a salir limpio tras este cuarto corte real de M3

    Bloque 5 ya ejecutado¶

    • nace CourtyardPrimaryStructureLedger.tsx para encapsular la rama del edificio principal en journey, founding y stronghold
    • DashboardCourtyard.tsx deja de renderizar directamente ese ledger y pierde otra superficie grande del pergamino sin cambiar comportamiento visible
    • el patio empieza ya a quedarse con ramas mas pequenas y reconocibles en vez de grandes bloques de JSX incrustados en el mismo componente
    • npm run build vuelve a salir limpio tras este quinto corte real de M3

    Bloque 6 ya ejecutado¶

    • nace CourtyardSelectedBuildingLedger.tsx para encapsular la ficha del edificio seleccionado, incluyendo resumen, controles de trabajadores, acciones de construccion o mejora y placeholders manuales
    • DashboardCourtyard.tsx deja de renderizar directamente la rama mas pesada que quedaba en el pergamino del patio
    • el patio gana otra bajada fuerte de peso y se acerca mucho mas a shell de composicion real, con la mayor parte de sus ledgers ya fuera
    • npm run build vuelve a salir limpio tras este sexto corte real de M3

    Bloque 7 ya ejecutado¶

    • nace CourtyardSettlementNamingLedger.tsx para encapsular la hoja de nombrado inicial del asentamiento, incluyendo copy, validacion del nombre, accion principal y estados de guardado
    • DashboardCourtyard.tsx deja de renderizar directamente esa rama residual del pergamino y se acerca todavia mas a una shell de composicion real
    • el patio pierde otra superficie autocontenida sin cambiar comportamiento visible y deja ya muy pocas ramas residuales grandes dentro del componente principal
    • npm run build vuelve a salir limpio tras este septimo corte real de M3

    Bloque 8 ya ejecutado¶

    • nace CourtyardTestingToolsOverlay.tsx para encapsular la superficie visual de las herramientas internas de QA, incluyendo acciones rapidas, cierre y ayudas de atajos
    • DashboardCourtyard.tsx deja de renderizar directamente ese overlay y se acerca un paso mas a una shell de composicion real
    • la logica de atajos, estado y acciones QA sigue viva en el patio por ahora, pero la superficie visual ya queda separada en un componente propio sin cambiar comportamiento
    • npm run build vuelve a salir limpio tras este octavo corte real de M3

    Bloque 9 ya ejecutado¶

    • DashboardCourtyard.tsx deja de leer world a traves de la fachada world-state.ts y pasa a importar desde world-model, world-selectors, world-commands y world-simulation
    • CourtyardOverviewLedger.tsx y useCourtyardLoggingMission.ts tambien dejan de depender de esa fachada temporal
    • el dashboard queda asi mucho mejor alineado con las fronteras abiertas en M1 y M2, sin cambiar comportamiento visible ni abrir mas componentes por abrir
    • npm run build vuelve a salir limpio tras este noveno corte real de M3

    Cierre validado¶

    M3 queda ya formalmente cerrado porque:

    1. DashboardSettingsRoom ya no mezcla la mayor parte de sus integraciones pesadas
    2. DashboardCourtyard ya no concentra seleccion local, wiring del slice activo, ledgers, nombrado inicial ni overlay de QA
    3. el dashboard ya no depende de la fachada temporal world-state
    4. npm run build siguio limpio durante los cortes finales
    5. el smoke test base final ha pasado sin regresiones visibles en onboarding, dashboard, persistencia base ni Dirigir tala

    El siguiente paso ya no pertenece a M3, sino a decidir como abrir M4 o si conviene una preparacion tecnica corta previa.

    Criterio de cierre de M3¶

    M3 se podra dar por cerrado cuando:

    • DashboardCourtyard.tsx ya no gobierne directamente el wiring temporal de Dirigir tala
    • DashboardSettingsRoom.tsx ya no mezcle shell visual con integraciones Tauri, dimensiones y acciones destructivas
    • ambos archivos queden como superficies de composicion razonables
    • npm run build siga limpio
    • el smoke test base siga pasando
    • toda pieza transicional nueva quede inventariada
