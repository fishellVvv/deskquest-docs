    Politica de legacy para M4¶

    Estado¶

    • estado: vigente
    • alcance: preparacion y ejecucion de M4

    Objetivo¶

    Fijar un criterio claro para decidir que legacy debe mantenerse y cual debe desaparecer a medida que DeskQuest abre M4.

    Esta politica existe para evitar dos problemas:

    • conservar compatibilidades antiguas "por si acaso" sin necesidad real
    • confundir el legacy de producto con el legacy transicional del refactor

    Decision vigente¶

    La linea actual del proyecto reconoce solo 0.1.0-beta.2 como referencia publica viva.

    Por tanto:

    • ya no se protege compatibilidad con datos, claves o eventos anteriores a 0.1.0-beta.2 si solo pertenecen a etapas de desarrollo o al rename desde test_game
    • test_game deja de considerarse una identidad soportada a futuro
    • el unico legacy que puede mantenerse por defecto es el que reduzca riesgo estructural del refactor activo o preserve una superficie publica todavia consumida dentro del repo

    Tipos de legacy¶

    1. Legacy de rename y compatibilidad pre-beta.2¶

    Incluye cosas como:

    • claves test_game.*
    • eventos o canales test_game.*
    • comandos antiguos mantenidos como fallback
    • migraciones de guardados o payloads de desarrollo previos a 0.1.0-beta.2

    Politica:

    • no debe mantenerse por inercia
    • puede eliminarse cuando toque el corte correspondiente
    • no se considera ya una obligacion operativa del proyecto

    2. Legacy de formato o datos antiguos¶

    Incluye:

    • loaders que aceptan snapshots viejos solo para no romper saves de desarrollo
    • normalizadores que siguen contemplando estructuras anteriores a la beta viva

    Politica:

    • no debe mantenerse si solo protege estados anteriores a 0.1.0-beta.2
    • puede retirarse durante M4 cuando el frente afectado se reordene

    3. Legacy de refactor¶

    Incluye:

    • wrappers transicionales como los backends de platform/world
    • cualquier adaptador temporal que evite romper demasiados consumidores a la vez mientras se mueve una frontera real

    Politica:

    • si puede mantenerse temporalmente
    • debe estar inventariado
    • debe indicar consumidores, motivo y milestone objetivo de desaparicion o cambio de forma

    Criterio para conservar legacy¶

    Una pieza legacy solo debe sobrevivir si cumple al menos una de estas condiciones:

    1. protege una superficie publica que todavia tenga consumidores vivos dentro del repo
    2. reduce claramente el riesgo del refactor activo
    3. permite una migracion progresiva ya planificada y documentada

    Criterio para eliminar legacy¶

    Una pieza legacy debe considerarse candidata a desaparecer si se mantiene solo por:

    • datos locales de desarrollo anteriores a 0.1.0-beta.2
    • el rename historico desde test_game
    • miedo generico a romper algo no identificado
    • duplicar dos identidades tecnicas cuando una ya no se considera soportada

    Implicacion directa para M4¶

    M4 no debe abrirse como limpieza horizontal de todo el repo, pero si debe usar esta politica como limite operativo.

    Eso significa:

    • el primer corte de M4 puede eliminar compatibilidad pre-beta.2 dentro del frente world si esa compatibilidad solo protege rename o datos de desarrollo antiguos
    • el mismo primer corte si debe conservar el legacy estrictamente transicional que haga falta para mover persistencia y sync hacia platform/world
    • otros stores como configuration, telemetry o themes/shared no deben mezclarse en ese primer corte salvo que bloqueen directamente la nueva frontera

    La ejecucion de M4 ya ha aplicado este criterio tambien fuera de world en los stores de configuration, telemetry y themes/shared, eliminando compatibilidad pre-beta.2 que ya no protegia nada vivo.

    Regla documental¶

    Cada vez que una pieza legacy siga viva debe quedar claro:

    • que protege realmente
    • quien la consume hoy
    • por que sigue existiendo
    • en que milestone deberia desaparecer o cambiar de forma

    Si no puede responderse eso con claridad, la pieza debe tratarse como candidata a eliminacion.
