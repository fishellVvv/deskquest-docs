    Milestone 2 - World runtime and session¶

    Estado¶

    • estado: cerrado
    • depende de: M1 ya cerrado
    • foco: world-runtime, world-session y descarga de WorldProvider
    • situacion actual: cerrado con build limpia, runtime explicitado, primer precedente correcto de game y smoke test final pasado

    Objetivo¶

    Hacer explicita la sesion viva del mundo y empezar a sacar el runtime del reino fuera de React.

    M2 no intenta rehacer toda la persistencia ni reorganizar el dashboard. Su objetivo es aclarar quien gobierna:

    • el arranque del mundo vivo
    • el heartbeat de sesion
    • la sincronizacion por reloj
    • la sincronizacion por telemetria
    • la suscripcion al snapshot persistido

    Mapa actual del runtime¶

    WorldProvider.tsx¶

    Hoy mezcla varias responsabilidades:

    1. bootstrap del snapshot inicial mediante beginWorldSession
    2. mantenimiento del heartbeat mediante startWorldSessionHeartbeat
    3. aplicacion unica del preset de desarrollo activo
    4. consumo de la API de world-runtime para snapshot, telemetria y reloj
    5. shell React del contexto y del snapshot visible del mundo

    world-storage.ts¶

    Hoy ya no es el backend unico del runtime. Su papel actual es transicional:

    • mantiene la superficie publica historica para comandos persistidos y helpers de snapshot
    • sigue ofreciendo compatibilidad para consumidores que aun no necesitan conocer el corte interno
    • ya no gobierna directamente el heartbeat ni la suscripcion del runtime vivo

    world-snapshot-storage.ts¶

    Hoy concentra:

    • carga, normalizacion y guardado del snapshot
    • compatibilidad legacy de almacenamiento
    • emision local y cross-window
    • suscripcion al snapshot compartido

    world-runtime-storage.ts¶

    Hoy concentra:

    • arranque en frio del runtime vivo
    • lectura y persistencia del heartbeat de runtime
    • sincronizacion por telemetria
    • sincronizacion por reloj

    Lectura del problema¶

    El problema de M2 no es solo que WorldProvider tenga varios useEffect.

    El problema real es que la sesion viva del mundo y su runtime siguen sin frontera explicita:

    • React sigue gobernando parte del runtime
    • el backend transicional de snapshot y runtime ya esta mas claro, pero sigue sin existir todavia una frontera final de platform/world/
    • la aplicacion ya tiene sesion y runtime explicitados, pero WorldProvider aun absorbe coordinacion React que deberia quedar mas delgada

    Que si entra en M2¶

    • crear el primer punto de entrada real en src/application/world/
    • hacer explicita la sesion del mundo
    • empezar a sacar el runtime del provider
    • adelgazar WorldProvider.tsx
    • preparar el terreno para el primer precedente correcto de game

    Que no entra en M2¶

    • partir DashboardCourtyard.tsx
    • partir DashboardSettingsRoom.tsx
    • rehacer por completo persistencia y sincronizacion
    • abrir platform/world/ sin una responsabilidad real
    • recolocar todavia todo Dirigir tala

    Primer corte recomendado¶

    1. abrir src/application/world/world-session.ts
    2. mover ahi la puerta de entrada conceptual de la sesion viva
    3. hacer que WorldProvider.tsx dependa de esa API en vez de hablar directamente con la sesion de world-storage
    4. mantener world-storage.ts como backend transicional del snapshot y el heartbeat mientras no exista aun platform/world/

    Bloque 1 ya ejecutado¶

    • nace world-session.ts como primer punto real en src/application/world/
    • WorldProvider.tsx deja de depender directamente de beginWorldRuntimeSession y touchWorldRuntimeSession
    • el heartbeat de sesion queda encapsulado ya por una API de aplicacion, aunque siga delegando transicionalmente en el backend de runtime
    • npm run build vuelve a salir limpio tras este primer corte de M2

    Bloque 2 ya ejecutado¶

    • nace world-runtime.ts como primera entrada real para suscripcion, reloj, telemetria y criterio de trabajo temporizado
    • WorldProvider.tsx deja de depender directamente de subscribeToWorldState, syncWorldStateFromClock, syncWorldStateFromTelemetry y hasPassiveWorldProduction
    • el provider sigue teniendo logica de preset, puente legacy y acciones del mundo, pero ya no absorbe directamente la API viva del runtime
    • npm run build vuelve a salir limpio tras este segundo corte de M2

    Bloque 3 ya ejecutado¶

    • nacen world-snapshot-storage.ts y world-runtime-storage.ts como separacion explicita del backend transicional
    • world-storage.ts deja de mezclar snapshot, heartbeat y comandos persistidos en un unico archivo y pasa a comportarse como superficie publica transicional mas estrecha
    • world-session.ts y world-runtime.ts ya consumen rutas mas precisas para sesion, reloj, telemetria y suscripcion
    • el mapa de M2 ya distingue con mas claridad React -> application/world -> backend transicional
    • npm run build vuelve a salir limpio tras este tercer corte de M2

    Bloque 4 ya ejecutado¶

    • nace world-actions.ts como entrada de aplicacion para replace, update, reset y las mutaciones persistidas principales del reino temprano
    • WorldProvider.tsx deja de depender directamente de domain/world/world-storage tambien para las acciones del mundo
    • world-storage deja ya de tocar React de forma directa y pasa a quedar consumido desde application/world y desde la referencia TypeScript curada
    • el provider sigue absorbiendo coordinacion React, presets y el puente legacy de settlementName, pero ya no es cliente directo de sesion, runtime ni acciones persistidas del backend historico
    • npm run build vuelve a salir limpio tras este cuarto corte de M2

    Bloque 5 ya ejecutado¶

    • nace world-legacy-settlement-name.ts como utilidad de aplicacion para la migracion transicional del nombre del asentamiento desde configuracion legacy hacia world
    • WorldProvider.tsx deja de contener la decision legacy de settlementName y pasa a invocarla como responsabilidad de aplicacion
    • el provider pierde otra pieza de logica especifica heredada sin abrir todavia platform/world
    • npm run build vuelve a salir limpio tras este quinto corte de M2

    Bloque 6 ya ejecutado¶

    • nace world-react-state.ts para hacer explicita la reconciliacion del snapshot visible del mundo dentro del shell React
    • WorldProvider.tsx deja de repetir la misma comparacion serializada en varios useEffect
    • el provider queda ya muy cerca de shell puro: consume API de aplicacion, mantiene el preset de desarrollo activo y expone el contexto React del mundo
    • npm run build vuelve a salir limpio tras este sexto corte de M2

    Bloque 7 ya ejecutado¶

    • logging-mission.ts deja de vivir dentro de features/game y pasa a domain/game como logica pura del primer slice activo
    • DashboardCourtyard.tsx, LoggingMissionScene.tsx y LoggingMissionLedger.tsx consumen ya esa logica como dominio, no como codigo embebido en features
    • con ello M2 deja ya sembrado el primer precedente correcto de game sin abrir todavia el split grande del dashboard
    • npm run build vuelve a salir limpio tras este septimo corte de M2

    Validacion actual¶

    • npm run build correcto tras los siete bloques tecnicos de M2
    • smoke test manual base vuelto a pasar tras el cierre del milestone

    Cierre de M2¶

    M2 ya queda cerrado porque:

    1. world-session, world-runtime y world-actions existen ya como puntos de entrada reales de application/world
    2. WorldProvider deja de gobernar directamente heartbeat, sincronizacion temporal, acciones persistidas y compatibilidad legacy del mundo
    3. el backend historico de world ya no vive en un unico archivo y distingue world-snapshot-storage de world-runtime-storage
    4. el primer precedente correcto de game queda ya sembrado al mover logging-mission.ts a domain/game
    5. npm run build ha seguido saliendo limpio tras todos los cortes tecnicos del milestone
    6. el smoke test manual final ha vuelto a pasar sin regresiones visibles

    Siguiente paso recomendado¶

    Con M2 ya cerrado, el siguiente paso no es abrir M3 automaticamente, sino decidir cual es el siguiente frente estructural real:

    1. abrir M3 si ya conviene partir las superficies grandes del dashboard
    2. preparar antes un bloque corto de decision/lectura si sigue habiendo dudas sobre el siguiente corte
    3. retomar despues el enlace entre la nueva base tecnica y la planificacion del midgame

    Criterio de cierre de M2¶

    M2 se considera cerrado porque:

    • la sesion del mundo ya es explicita
    • el runtime del mundo ya no esta gobernado desde WorldProvider
    • React ya no absorbe heartbeat, timers ni sincronizacion base del mundo
    • el primer precedente correcto de game ya no depende de un patron torcido
    • build y smoke test han seguido limpios
    • las piezas transicionales de M2 siguen inventariadas
