    Preparacion de M4 - persistencia, sync y legacy¶

    Estado¶

    • estado: cerrado
    • milestone asociado: M4
    • abre milestone: no

    Objetivo¶

    Traducir M4 a un frente tecnico real antes de abrirlo.

    Esta preparacion existe para responder tres preguntas:

    1. que parte de persistencia, sync y legacy sigue realmente dispersa
    2. cual debe ser el primer corte util de M4
    3. que cosas no conviene mezclar todavia aunque tambien usen localStorage, BroadcastChannel o claves legacy

    Politica vigente de legacy¶

    La preparacion de M4 ya no parte de una compatibilidad abierta con todo el pasado del proyecto.

    La referencia viva es 0.1.0-beta.2.

    Por tanto:

    • el legacy de rename y de datos anterior a 0.1.0-beta.2 deja de ser una obligacion del producto
    • test_game ya no se considera una identidad soportada a futuro
    • solo debe mantenerse el legacy que reduzca riesgo estructural del refactor activo o preserve una superficie publica todavia consumida

    La politica detallada queda fijada en:

    • Politica de legacy para M4

    Lectura actual¶

    La revision del repo deja una foto bastante clara.

    1. El nucleo real de M4 sigue estando en world¶

    Hoy la deuda mas estructural no esta en todos los stores a la vez, sino en que la persistencia y la sincronizacion del mundo siguen viviendo en modulos de domain/world aunque ya funcionan de facto como backend de plataforma.

    Piezas implicadas:

    • world-storage.ts
    • world-snapshot-storage.ts
    • world-runtime-storage.ts
    • world-actions.ts
    • world-runtime.ts
    • world-session.ts

    La conclusion es directa:

    • application/world ya existe y coordina
    • domain/world ya esta razonablemente limpio en modelo/selectores/comandos/simulacion
    • lo siguiente natural es sacar de domain/world la parte de almacenamiento y sincronizacion que ya no pertenece ahi

    2. Hay otros stores con patrones parecidos, pero no deben abrir el milestone¶

    La preparacion encuentra repeticiones reales en:

    • configuration-store.ts
    • telemetry-store.ts
    • active-dimension-storage.ts
    • installed-dimensions-events.ts
    • browser-storage-migration.ts

    Todos usan, en mayor o menor medida:

    • localStorage
    • eventos de ventana
    • BroadcastChannel
    • claves legacy test_game.*

    Pero hoy no todos tienen el mismo riesgo arquitectonico.

    La recomendacion es:

    • no abrir M4 como una limpieza masiva de todos los stores
    • usar estos modulos como referencia de patrones repetidos
    • dejar una posible unificacion transversal para despues de estabilizar el camino de world
    • no mezclar en el primer corte la limpieza completa de configuration, telemetry o themes/shared aunque ahi tambien queden restos test_game.*

    3. La referencia publica aun conserva superficie historica¶

    typescript-api.ts sigue reexportando desde:

    • world-state.ts
    • world-storage.ts

    Eso no bloquea M4, pero si obliga a tratar compatibilidad publica con cuidado.

    La implicacion practica es:

    • el primer corte de M4 no deberia romper de golpe esa superficie
    • conviene conservar wrappers controlados mientras se mueve la frontera real

    Recomendacion de apertura para M4¶

    M4 deberia abrirse con foco estrecho en world persistence + runtime sync.

    No recomendaria empezar por:

    • refactor transversal de todos los stores browser
    • unificacion prematura de BroadcastChannel
    • limpieza global de todas las claves legacy del repo en el mismo bloque
    • rediseño de la referencia TypeScript completa

    Si recomendaria empezar por:

    1. abrir src/platform/world
    2. mover ahi el backend de snapshot del mundo
    3. mover ahi el backend de heartbeat/runtime sync del mundo
    4. dejar world-storage.ts como wrapper transicional mas fino
    5. redirigir application/world para que dependa de platform/world y no de domain/world para almacenamiento/sync
    6. aprovechar ese mismo frente para empezar a retirar compatibilidad pre-beta.2 del camino de persistencia/sync de world cuando ya no proteja nada vivo

    Primer corte recomendado de M4¶

    El primer bloque real que deja preparado esta revision es este:

    1. crear src/platform/world/
    2. mover o recrear ahi:
    3. world-snapshot-storage
    4. world-runtime-storage
    5. actualizar:
    6. world-runtime.ts
    7. world-session.ts
    8. world-actions.ts
    9. dejar world-storage.ts como superficie transicional inventariada, ya sin responsabilidades nuevas
    10. validar con npm run build

    Cosas que dejaria fuera del primer corte¶

    • mover configuration-store
    • mover telemetry-store
    • tocar active-dimension-storage
    • tocar installed-dimensions-events
    • extraer un helper generico de sync multi-store
    • limpiar toda la referencia typescript-api.ts

    Nada de eso queda descartado.

    Solo se recomienda no mezclarlo con la apertura del milestone porque ensancharia demasiado el frente.

    Riesgos especificos¶

    • mover persistencia de world a platform/ sin wrapper temporal podria romper demasiado codigo de golpe
    • tocar a la vez world, configuration, telemetry y themes aumentaria el riesgo sin aclarar mejor el contrato final
    • typescript-api.ts puede exigir una capa de compatibilidad temporal mientras se recoloca la frontera publica

    Cierre de la preparacion¶

    Esta preparacion deja recomendado:

    • no abrir M4 a ciegas
    • no abrirlo como cleanup horizontal de todo el repo
    • abrirlo, cuando toque, con un primer corte corto centrado en platform/world
    • usar desde el primer corte la politica de legacy para no seguir arrastrando compatibilidad test_game o datos pre-beta.2 sin motivo real

    El siguiente paso ya no es explorar mas, sino decidir si se aprueba esa apertura o si antes se quiere otra comprobacion puntual.

    Bloque 1 - platform/world (completado)¶

    El primer corte real de M4 ya queda ejecutado.

    Se completa asi:

    • nace src/platform/world
    • world-snapshot-storage.ts y world-runtime-storage.ts pasan a vivir ya en platform/world
    • world-session.ts y world-runtime.ts ya dependen de esa nueva frontera
    • world-storage.ts se mantiene como wrapper transicional
    • el camino de persistencia y runtime de world deja de aceptar varias compatibilidades pre-beta.2, incluyendo claves test_game.world, test_game.world.runtime_heartbeat y formas viejas de guardado del mundo

    Validacion:

    • npm run build limpio

    Implicacion:

    • M4 deja ya de estar solo preparado y pasa a estar realmente abierto
    • la nueva frontera platform/world ya existe de verdad en codigo

    Bloque 2 - adelgazado de world-state (completado)¶

    El segundo corte estrecho de M4 ya queda ejecutado.

    Se completa asi:

    • IdleStructureArt.tsx deja de importar WorldPhase desde world-state
    • typescript-api.ts deja de reexportar el bloque principal de world desde world-state.ts
    • la referencia pasa ya a salir de:
    • world-model.ts
    • world-commands.ts
    • world-selectors.ts
    • world-simulation.ts
    • world-state.ts queda ya como shim puro sin consumidores vivos dentro de src

    Validacion:

    • npm run build limpio

    Implicacion:

    • world-state queda practicamente listo para desaparecer
    • el siguiente corte ya puede centrarse en world-storage o en retirar la propia fachada sin mezclar mas frentes

    Bloque 3 - retirada de world-state (completado)¶

    El tercer corte estrecho de M4 ya queda ejecutado.

    Se completa asi:

    • world-state.ts desaparece del codigo vivo
    • la vista general de arquitectura deja de presentarlo como fachada activa
    • el inventario transicional deja de contarlo entre las entradas vivas
    • la referencia historica del milestone pasa a marcarlo como retirado en M4

    Validacion:

    • npm run build limpio

    Implicacion:

    • world deja ya de depender de esa fachada historica
    • el siguiente corte puede centrarse por fin en adelgazar world-storage

    Bloque 4 - estrechamiento de world-storage (completado)¶

    El cuarto corte estrecho de M4 ya queda ejecutado.

    Se completa asi:

    • world-storage.ts deja de reexportar snapshot y runtime:
    • loadStoredWorldState
    • replaceWorldState
    • saveWorldState
    • subscribeToWorldState
    • syncWorldStateFromClock
    • syncWorldStateFromTelemetry
    • beginWorldRuntimeSession
    • touchWorldRuntimeSession
    • world-actions.ts pasa a tomar replaceWorldState directamente desde world-snapshot-storage.ts
    • typescript-api.ts conserva ya solo las acciones persistidas de alto nivel del mundo desde world-storage.ts

    Validacion:

    • npm run build limpio

    Implicacion:

    • world-storage deja de actuar como puerta mixta de acciones, snapshot y runtime
    • la superficie historica de world queda ya mucho mas cerca de un wrapper fino
    • el siguiente corte puede centrarse en decidir que parte de esa fachada sigue teniendo sentido como API residual y cual debe reencaminarse ya hacia application/world o platform/world sin tener dos wrappers superpuestos en el mismo frente

    Bloque 5 - retirada de world-storage (completado)¶

    El quinto corte estrecho de M4 ya queda ejecutado.

    Se completa asi:

    • world-actions.ts absorbe ya la logica residual que seguia viva en world-storage.ts
    • typescript-api.ts deja de reexportar las acciones persistidas del mundo desde domain/world y pasa a hacerlo ya desde application/world
    • world-storage.ts sale del codigo vivo al quedarse sin consumidores dentro de src

    Validacion:

    • npm run build limpio

    Implicacion:

    • el frente world deja ya de mantener wrappers historicos vivos dentro de domain/world
    • la cadena activa queda ya mucho mas clara: domain/world para reglas puras, application/world para orquestacion y platform/world para persistencia y sync
    • el siguiente paso de M4 pasa a ser decidir si el milestone remata ahora el cleanup corto de legacy pre-beta.2 fuera de world o si ya puede entrar en validacion final

    Bloque 6A - cleanup de legacy en configuration y telemetry (completado)¶

    El siguiente corte estrecho de M4 ya queda ejecutado sobre dos stores aislados que todavia mantenian compatibilidad pre-beta.2.

    Se completa asi:

    • configuration-store.ts deja de aceptar test_game.configuration
    • telemetry-store.ts deja de aceptar test_game.telemetry
    • telemetry deja tambien de contemplar el formato antiguo con clickCount/keypressCount

    Validacion:

    • npm run build limpio

    Implicacion:

    • el cleanup corto de legacy pre-beta.2 ya no es solo una politica de world
    • el siguiente paso natural de M4 pasa a ser ejecutar el bloque equivalente sobre themes/shared antes de lanzar la validacion final del milestone

    Bloque 6B - cleanup de legacy en themes/shared (completado)¶

    El bloque equivalente de M4 sobre dimensiones compartidas ya queda ejecutado.

    Se completa asi:

    • active-dimension-storage.ts deja de aceptar test_game.active_dimension y test_game.active_theme
    • installed-dimensions-events.ts deja de aceptar test_game.installed_dimensions_changed y test_game.theme_library_reload
    • desaparece tambien el canal legacy test-game-theme-library

    Validacion:

    • npm run build limpio

    Implicacion:

    • el cleanup corto de legacy pre-beta.2 queda ya ejecutado tambien fuera de world, configuration y telemetry
    • M4 queda preparado para validacion final sin seguir arrastrando identidad test_game en los stores browser activos del runtime

    Cierre de M4¶

    La validacion final de M4 ya queda completada.

    Se confirma:

    • npm run build habia seguido saliendo limpio durante todos los cortes del milestone
    • el smoke test final ya ha pasado sin regresiones visibles
    • domain/world queda ya libre de wrappers historicos vivos
    • application/world y platform/world quedan ya explicitados como fronteras reales
    • el cleanup corto de compatibilidad pre-beta.2 queda aplicado tambien fuera de world

    Con ello, M4 deja de estar activo y pasa a considerarse cerrado.

    El siguiente frente ya no pertenece al nucleo del milestone, sino a un bloque residual mas corto:

    • Preparacion de M4b - cleanup residual de legacy
