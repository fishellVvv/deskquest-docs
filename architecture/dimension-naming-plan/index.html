    Plan de nomenclatura y limpieza del sistema de dimensiones¶

    Objetivo¶

    Fijar un vocabulario estable y un mapa de limpieza por fases para que el proyecto deje de mezclar skin, theme, library y dimension sin perder claridad interna ni romper la arquitectura que ya funciona.

    Este documento no ejecuta aun los renombres delicados. Su funcion es dejar cerrado:

    • que termino usamos en cada capa
    • que modulos conviene mantener tal y como estan
    • que modulos conviene renombrar o dividir
    • en que orden debe hacerse la limpieza

    Glosario oficial¶

    Dimension¶

    Concepto de producto visible para el usuario.

    Una dimension es algo que el usuario puede:

    • anadir
    • instalar
    • ver en una lista
    • viajar
    • quitar

    Ejemplos:

    • Medieval
    • Base lunar
    • Fortaleza de vapor

    Dimension integrada¶

    Dimension que vive dentro del codigo nativo y no puede desaparecer.

    Regla actual:

    • la unica dimension integrada es builtin:medieval

    Dimension instalada¶

    Dimension ya importada e instalada dentro de la carpeta runtime interna de la app.

    Ruta conceptual:

    <app-data>/installed-dimensions/<dimension-id>/

    Paquete .dimension¶

    Formato de intercambio y distribucion.

    El archivo .dimension no es la dimension en ejecucion, sino el paquete que se importa para convertirla en una dimension instalada.

    Demo oficial¶

    Paquete mantenido por el propio proyecto para documentar el sistema y servir de referencia a modders.

    Regla actual:

    • io.github.fishellvvv.lunar-base.dimension es la demo oficial

    Registro runtime de dimensiones¶

    Coleccion de dimensiones que la app conoce en memoria en un momento dado:

    • builtin:medieval
    • dimensiones instaladas

    Su responsabilidad es resolver ids, fallback y orden de disponibilidad.

    Tema resuelto¶

    Resultado interno de aplicar:

    • dimension activa
    • fallback a builtin:medieval
    • idioma
    • tokens
    • textos
    • assets

    Este termino sigue teniendo sentido dentro de la capa visual y de render.

    Por eso, en esta fase, no conviene renombrar a ciegas todo lo que hoy se llama theme.

    Fallback absoluto¶

    Si algo falla o falta, la app debe volver siempre a:

    • builtin:medieval

    Limites entre capas¶

    Capa 1. Producto y UX¶

    Aqui siempre debemos hablar de:

    • dimension
    • dimension instalada
    • archivo .dimension
    • anadir
    • viajar
    • quitar

    Esta capa incluye:

    • copys visibles
    • botones
    • listas
    • mensajes de importacion
    • documentacion para usuario/modder

    Capa 2. Paquete e importacion¶

    Aqui debemos hablar de:

    • dimension package
    • dimension archive
    • installed dimensions

    Esta capa incluye:

    • validacion del manifest
    • importacion desde archivo
    • inspeccion previa
    • instalacion en carpeta interna
    • eliminacion

    Capa 3. Resolucion visual interna¶

    Aqui todavia es razonable mantener theme mientras siga describiendo el tema ya resuelto que usan el idle y el dashboard.

    Esta capa incluye:

    • ThemeProvider
    • ThemeDefinition
    • applyThemeToDocument
    • registro runtime que mezcla tokens, lexicon y assets

    Regla:

    • dimension para la identidad del paquete y del producto
    • theme para el resultado visual resuelto

    Decisiones de limpieza¶

    Modulos que mantenemos por ahora¶

    Estos nombres siguen siendo aceptables en esta fase:

    • src/app/providers/ThemeProvider.tsx
    • src/themes/shared/theme-definition.ts
    • src/themes/shared/runtime-theme-registry.ts
    • src/themes/shared/apply-theme.ts

    Motivo:

    • describen la capa visual resuelta
    • renombrarlos ahora aportaria poco y moveria demasiado codigo

    Modulos que mantenemos como compatibilidad temporal¶

    • src/themes/shared/theme-registry.ts

    Motivo:

    • hoy sigue funcionando como adaptador fino sobre el registro runtime
    • conviene revisarlo despues de la limpieza principal

    Objetivo posterior:

    • decidir si se elimina o si se convierte en un alias muy pequeno

    Renombres ejecutados en Fase B¶

    1. src/themes/shared/skin-package.ts¶

    Resultado:

    • src/themes/shared/dimension-package.ts

    Motivo:

    • describe el contrato del paquete .dimension
    • ya no deberia usar skin como termino principal

    2. src/themes/shared/external-skin-loader.ts¶

    Resultado:

    • se divide en:
    • src/themes/shared/installed-dimensions-client.ts
    • src/themes/shared/dimension-package-client.ts

    Responsabilidad de cada uno:

    • installed-dimensions-client.ts
    • cargar dimensiones instaladas
    • instalar demo oficial
    • quitar dimension instalada
    • dimension-package-client.ts
    • inspeccionar archivo .dimension
    • importar archivo .dimension

    3. src/themes/shared/theme-library-storage.ts¶

    Resultado:

    • src/themes/shared/installed-dimensions-events.ts

    Motivo:

    • ya no sincroniza una biblioteca escaneable
    • sincroniza cambios sobre dimensiones instaladas

    4. src/themes/shared/theme-storage.ts¶

    Resultado:

    • src/themes/shared/active-dimension-storage.ts

    Motivo:

    • esta capa guarda la seleccion de la dimension activa del usuario
    • aqui pesa mas el lenguaje de producto que el de tema resuelto

    Renombre ejecutado en Fase C¶

    src-tauri/src/skins.rs¶

    Resultado:

    • src-tauri/src/dimensions.rs

    Motivo:

    • la capa nativa ya no trabaja con una carpeta generica de skins
    • trabaja con importacion, instalacion y lectura de dimensiones

    Renombres de tipos y comandos¶

    Rust¶

    • ExternalSkinScanSnapshot -> InstalledDimensionsSnapshotPayload
    • ExternalSkinDirectoryPackage -> DimensionPackagePayload
    • ExternalSkinLocaleFile -> DimensionLocaleFilePayload
    • ExternalSkinAssetFiles -> DimensionAssetFilesPayload
    • ExternalSkinScanError -> InstalledDimensionLoadErrorPayload
    • SkinArchiveInspection -> DimensionPackageInspection
    • InstalledSkinImportResult -> InstalledDimensionImportResult

    Comandos Tauri¶

    • scan_installed_skin_packages -> load_installed_dimensions_snapshot
    • install_official_skin_package -> install_official_dimension
    • inspect_skin_archive -> inspect_dimension_package
    • install_skin_archive -> install_dimension_package
    • remove_installed_skin_package -> remove_installed_dimension
    • read_external_skin_svg_asset -> read_dimension_svg_asset

    Frontend¶

    • loadExternalSkinLibrarySnapshot -> loadInstalledDimensionsSnapshot
    • installOfficialSkinPackage -> installOfficialDimension
    • inspectSkinArchive -> inspectDimensionPackage
    • installSkinArchive -> installDimensionPackage
    • removeInstalledSkinPackage -> removeInstalledDimension

    Orden de ejecucion recomendado¶

    Fase A. Glosario y auditoria¶

    Estado:

    • completada en este documento

    Entregable:

    • vocabulario oficial
    • mapa de modulos a mantener y renombrar

    Fase B. Capa frontend de importacion e instalacion¶

    Objetivo:

    • limpiar la capa de clientes TS sin tocar aun la resolucion visual

    Estado:

    • completada

    Resultado:

    • dimension-package.ts sustituye a skin-package.ts
    • installed-dimensions-client.ts y dimension-package-client.ts sustituyen a external-skin-loader.ts
    • installed-dimensions-events.ts sustituye a theme-library-storage.ts
    • active-dimension-storage.ts sustituye a theme-storage.ts
    • ThemeProvider y la UI de configuracion ya consumen la nueva nomenclatura

    Fase C. Capa nativa de dimensiones¶

    Objetivo:

    • alinear Rust y los comandos Tauri con el modelo final

    Estado:

    • completada

    Resultado:

    • skins.rs pasa a dimensions.rs
    • los structs nativos adoptan ya nomenclatura de dimension
    • los comandos Tauri dejan de hablar de skin y quedan alineados con los clientes TS
    • el idle lee SVG externos por read_dimension_svg_asset

    Fase D. UI y copy¶

    Objetivo:

    • eliminar restos visibles de skin, library o naming heredado

    Estado:

    • completada

    Resultado:

    • el copy visible del panel se alinea con dimension e instalada
    • la especificacion principal se consolida como DIMENSION_PACKAGE_FORMAT
    • existe una guia rapida de modding
    • existe un checklist QA especifico del sistema de dimensiones

    Fase E. QA manual¶

    Objetivo:

    • cerrar la limpieza con una verificacion real de producto

    Checklist minima:

    • importar una dimension nueva
    • sobrescribir una dimension con el mismo id
    • avisar cuando coincida el nombre visible
    • quitar una dimension inactiva
    • quitar una dimension activa y volver a Medieval
    • cerrar y reabrir la app
    • comprobar que lo instalado sigue ahi
    • comprobar que idle y dashboard siguen alineados

    Regla para los siguientes cambios¶

    En adelante:

    • no renombrar por estetica
    • renombrar solo cuando el nombre este describiendo mal la capa
    • mantener theme donde siga significando "tema visual resuelto"
    • usar dimension donde hablemos de identidad, instalacion, importacion o UX

    Estado actual tras Fase C¶

    La capa de importacion e instalacion queda ahora dividida asi:

    • dimension-package.ts
    • contrato del paquete .dimension
    • dimension-package-client.ts
    • inspeccion e importacion de archivos .dimension
    • installed-dimensions-client.ts
    • carga, instalacion oficial y eliminacion de dimensiones instaladas
    • active-dimension-storage.ts
    • persistencia de la dimension activa
    • installed-dimensions-events.ts
    • sincronizacion entre ventanas cuando cambian las dimensiones instaladas
    • src-tauri/src/dimensions.rs
    • carga de dimensiones instaladas
    • instalacion oficial
    • inspeccion e importacion de paquetes .dimension
    • eliminacion de dimensiones instaladas
    • lectura segura de SVGs de dimension

    El siguiente frente correcto ya no es la nomenclatura base, sino la limpieza final de copy, docs y checklist QA.
