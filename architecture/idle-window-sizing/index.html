    Calibracion del idle raster¶

    Esta nota recoge un bug especialmente enganoso del idle cuando se paso de arte vectorial/placeholder a un PNG raster limpio como src/themes/medieval/assets/idle/core/core-settlement.png.

    Sintoma¶

    La ventana del idle se declaraba a 320x160 y el PNG tambien media 320x160, pero en runtime el castillo seguia viendose cortado por arriba e izquierda.

    El problema:

    • aparecia en varias pantallas, no solo en portatiles con escalado
    • podia "arreglarse" temporalmente al mover el idle entre monitores
    • volvia a romperse al cerrar y abrir la app

    Eso hacia pensar primero en DPI o en un PNG mal exportado, pero ninguna de esas dos explicaciones era la real.

    Diagnostico correcto¶

    El error no estaba en el asset.

    El PNG media correctamente 320x160 y su contenido visible ni siquiera tocaba el borde superior o izquierdo del archivo. El desajuste real estaba en la presentacion de ventana:

    • el proyecto pedia un tamano de ventana en pixeles logicos
    • luego intentaba validar el resultado usando innerSize() y outerSize()
    • pero lo que determina si el arte cabe o no dentro del idle es el viewport CSS real del WebView: window.innerWidth y window.innerHeight

    En otras palabras:

    • tamano de ventana Tauri
    • tamano interno del WebView
    • tamano CSS efectivo del viewport

    no estaban alineados del todo, y por eso un 320x160 podia no equivaler al otro.

    Solucion adoptada¶

    La calibracion del idle se hace ahora contra el viewport CSS real.

    El flujo correcto es:

    1. pedir un tamano logico inicial de ventana a partir del layout activo
    2. mostrar la ventana
    3. medir window.innerWidth y window.innerHeight
    4. comparar ese viewport con el tamano deseado de la escena
    5. corregir el tamano de la ventana hasta que el viewport CSS coincida con el layout del idle
    6. recolocar la ventana y hacer una segunda pasada de asentado

    Regla tecnica que queda fijada¶

    Para arte raster del idle, la referencia canonica no debe ser:

    • solo outerSize()
    • ni solo innerSize()

    La referencia canonica debe ser:

    • window.innerWidth
    • window.innerHeight

    porque eso es lo que finalmente consume React/CSS al pintar el arte.

    Consecuencias practicas¶

    • el idle ya puede mostrar correctamente un PNG raster 1:1
    • el mismo mecanismo se adapta al cambiar entre castillo medieval, carreta o base lunar
    • futuras overlays del idle como banderas, humo, iconos o hitboxes pueden seguir usando el mismo sistema de coordenadas del layout

    Interaccion del idle¶

    Una vez estabilizado el sizing raster, aparecio un segundo problema de escritorio: aunque el idle fuera transparente visualmente, su ventana seguia ocupando un rectangulo interactivo completo y podia bloquear botones del dashboard visibles a traves de pixeles transparentes del arte.

    La solucion adoptada fija otra regla importante:

    • la ventana visual del idle debe ser click-through
    • la interaccion no vive dentro del arte
    • las zonas clicables se materializan como hotspots explicitos

    En la primera iteracion, eso se resuelve con:

    1. una ventana principal visual del idle que usa setIgnoreCursorEvents(true)
    2. mini ventanas transparentes e independientes para cada hotspot
    3. sincronizacion de esas mini ventanas contra el viewport real ya calibrado del idle

    Esto evita dos problemas a la vez:

    • los pixeles transparentes dejan de bloquear clicks del dashboard
    • la interaccion futura no depende de hacer hit-testing por pixel sobre el PNG

    Regla de arquitectura que queda fijada¶

    Para el idle, arte e interaccion deben tratarse como capas distintas:

    • arte: una ventana visual, transparente y no interactiva
    • hotspots: ventanas pequenas, alineadas con zonas concretas del layout

    Esto deja una base mas limpia para:

    • puerta del asentamiento
    • campanas de audio
    • banderas o marcadores de bonus
    • futuras capas decorativas superpuestas

    Bandeja del sistema¶

    Una vez separado el idle entre arte visual y hotspots, aparece otra regla de ergonomia de escritorio:

    • el idle no debe ensuciar la barra de tareas
    • pero la app si debe seguir teniendo una presencia nativa recuperable

    La solucion adoptada es:

    • idle: sin acceso propio en barra de tareas
    • dashboard: con acceso normal en barra de tareas
    • tray: icono persistente con menu nativo

    El menu de bandeja resuelve por ahora dos acciones:

    • Mostrar / ocultar
    • Cerrar aplicacion

    La accion Mostrar / ocultar no afecta solo a la ventana visual principal:

    • oculta o restaura la ventana main
    • oculta o restaura los hotspots del idle
    • recuerda si el dashboard estaba visible antes de ocultar
    • si lo estaba, lo devuelve junto al resto al restaurar

    Con esto se mantiene una regla estable de producto:

    • el idle sigue comportandose como presencia discreta de escritorio
    • el dashboard sigue comportandose como ventana operativa del reino
    • la bandeja del sistema pasa a ser el punto nativo de control de la aplicacion

    Failsafe de visibilidad del idle¶

    Despues de estabilizar la apertura del dashboard, aparecio un bug distinto en el idle: la ventana nativa llegaba a abrirse, aparecia incluso en la barra de tareas y Windows la consideraba visible, pero la escena no llegaba a pintarse.

    El sintoma era especialmente enganoso:

    • el proceso test_game seguia vivo
    • la ventana tenia coordenadas y tamano validos
    • al capturar la zona de pantalla correspondiente, solo se veia el fondo del escritorio
    • el cursor tampoco cambiaba porque la ventana visual del idle es click-through

    Diagnostico¶

    La cadena de "ready" del idle dependia de tres senales:

    • ventana asentada (isWindowPrepared)
    • fuentes listas (areFontsReady)
    • arte cargado (isArtworkReady)

    Si una de esas senales quedaba colgada, la opacidad visual del idle no subia nunca y la ventana quedaba viva pero totalmente invisible.

    Solucion adoptada¶

    Se anade un failsafe temporal en IdleShell:

    1. el idle sigue intentando revelarse solo cuando todo esta listo
    2. pero si tras un pequeno margen de seguridad la ventana ya tiene un viewport valido y la cadena de ready no ha terminado, se fuerza igualmente la aparicion de la escena

    Esto deja una regla practica muy util:

    • es mejor mostrar el idle con una degradacion elegante
    • que dejar una ventana nativa viva pero invisible para siempre

    Cuadro de emergencia¶

    Si vuelve a ocurrir algo parecido:

    1. comprobar primero si la ventana existe de verdad y donde esta colocada
    2. capturar esa zona de pantalla para confirmar si hay render o transparencia total
    3. revisar si la cadena isWindowPrepared + areFontsReady + isArtworkReady esta bloqueando la opacidad

    Con este failsafe, ese tipo de cuelgue ya no deberia volver a dejar la app sin feedback visible.

    Archivos implicados¶

    • src/features/idle/hooks/useIdleWindowPresentation.ts
    • src/features/idle/components/IdleShell.tsx
    • src/features/idle/components/IdleStructureArt.tsx
    • src/shared/styles/global.css
    • src/themes/medieval/idle-layout.ts

    Que no hacer¶

    Para este caso se descartan como solucion buena:

    • forzar el PNG a encajar con object-fit
    • deformar el arte para adaptarlo a la ventana
    • meter margenes internos falsos alrededor del asset
    • compensar el problema recortando o reexportando el PNG

    Todo eso tapa el sintoma, pero rompe la alineacion futura entre arte, hitboxes y overlays.
