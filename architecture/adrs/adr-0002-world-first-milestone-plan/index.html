    ADR-0002 - DeskQuest executes the architecture refactor with a world-first milestone plan¶

    • Status: Accepted
    • Date: 2026-03-27
    • Deciders: DeskQuest team
    • Scope: Execution strategy for the architecture refactor
    • Supersedes: informal sequencing proposed in the initial architecture review
    • Tags: execution, refactor, milestones, world-first

    Context¶

    ADR-0001 establishes the target architecture for DeskQuest as a modular layered monolith.

    The remaining question is not the architectural direction itself, but how to execute the refactor without destabilizing the product or opening too many fronts at once.

    The codebase review and team feedback converge on the following risk reading:

    • world currently concentrates too much of the game truth
    • part of the runtime lifecycle is hidden inside providers
    • DashboardCourtyard and DashboardSettingsRoom absorb responsibilities partly because the core remains too dense
    • the first active gameplay slice already creates a precedent that should be relocated before more slices grow the same way

    The team considers that the execution order matters as much as the target model.

    Decision¶

    DeskQuest executes the architecture refactor with a world-first milestone plan.

    This means:

    • the architecture target defined in ADR-0001 remains stable
    • the execution sequence is treated as the current operating strategy, not as an immutable product truth
    • the first structural priority is to make world more legible, separable, and extractable
    • runtime extraction follows that clarification
    • large dashboard surfaces are decomposed after the world/runtime responsibilities become clearer
    • infrastructure normalization is delayed until it supports clearly identified contracts instead of being reordered preemptively

    Important clarification¶

    world-first does not mean “perfect world before anything else”.

    The objective of the first structural milestone is: - to make world more legible - to separate the most important internal responsibilities - to prepare future extractions safely

    It is not to exhaust all structural debt inside world before progressing.

    Operational principles¶

    The execution strategy follows these principles:

    • no rewrite
    • no opening all phases in parallel
    • no additional business logic inside frozen hotspots except what is strictly needed for extraction
    • no new architectural layers without real responsibility
    • each milestone must leave the game stable and testable
    • documentation must evolve with the refactor, not after it
    • temporary wrappers and facades are allowed only as controlled transition devices

    Milestone plan¶

    Milestone 0 - Alignment and safety net¶

    Goal¶

    • approve the architectural frame
    • freeze critical files
    • define minimum operating rules
    • establish a repeatable manual smoke test
    • register a baseline of file size, responsibility hotspots, and current behavior

    Expected outcome¶

    • the team knows what is in scope and what is not
    • critical files stop growing while extraction preparation begins
    • the refactor gains a controlled baseline

    Mandatory rules¶

    The following rules apply from this milestone onward:

    • DashboardCourtyard is frozen except for:
    • hotfixes
    • temporary wiring needed for extraction
    • removal of code that has already been moved out
    • DashboardSettingsRoom is frozen under the same rule
    • WorldProvider does not accept new business logic beyond extraction support
    • no new gameplay simulation is added directly into features/ without explicit architectural review
    • new environment integrations must go through platform/ once that route exists
    • new world-level use cases must go through application/ once that route exists

    Milestone 1 - Minimal boundaries + world core¶

    Goal¶

    • create src/application/ and src/platform/ in a minimal and useful way
    • avoid empty architecture folders and bureaucratic abstractions
    • begin immediate decomposition of world

    Internal order¶

    1. create minimal boundaries with real responsibility or clearly transitional wrappers
    2. divide world into:
    3. model/types
    4. selectors
    5. commands
    6. simulation
    7. preserve transitional facades only where they reduce breakage risk meaningfully

    Expected outcome¶

    • the world core becomes more legible
    • future growth no longer depends on a single dense block
    • the architecture starts to exist in code, not only in documents

    Clarification¶

    This milestone does not aim to “finish world forever”. It aims to make world understandable enough to support the next extractions safely.

    Milestone 2 - World runtime and first correct game precedent¶

    Goal¶

    • extract runtime lifecycle outside WorldProvider
    • make the live world session explicit
    • create the first active gameplay module aligned with the target architecture

    Internal order¶

    1. extract world-runtime and world-session
    2. slim WorldProvider
    3. move the real gameplay logic of Dirigir tala out of features/

    Expected outcome¶

    • the ownership of the world lifecycle becomes explicit
    • React providers stop acting as hidden runtime engines
    • game receives a first correct precedent separating simulation, orchestration, and presentation

    Milestone 3 - Large dashboard surfaces¶

    Goal¶

    • split DashboardCourtyard
    • split DashboardSettingsRoom
    • leave both as composition shells

    Expected outcome¶

    • the main dashboard surfaces stop concentrating excessive responsibility
    • composition becomes more visible
    • the UI becomes easier to maintain without redistributing logic arbitrarily

    Milestone 4 - Repositories, synchronization, and legacy isolation¶

    Goal¶

    • normalize persistence and sync patterns
    • encapsulate legacy compatibility
    • consolidate dependency rules in the new structure

    Expected outcome¶

    • storage and sync patterns stop being scattered repetitions
    • legacy contracts are localized
    • structural integration debt is reduced

    Guardrail¶

    This milestone is not an excuse to start unrelated infrastructure cleanups early. Infrastructure is only advanced earlier when it directly unlocks world-runtime or world-session. It must not create a parallel “let’s tidy sync now” mini-phase for convenience.

    Milestone 5 - Documentation consolidation and midgame readiness¶

    Goal¶

    • leave a living architecture record
    • close minimum ADRs and architectural notes
    • prepare the next product systems so they do not fall into the same structural patterns

    Expected outcome¶

    • architecture stops depending on tacit knowledge
    • future systems enter a clearer structure
    • the project gains a safer foundation for midgame growth

    Documentation policy by milestone¶

    Documentation is not deferred to the end. The following minimum expectation applies:

    • Milestone 0: architectural frame, operating rules, baseline, and hotspot inventory
    • Milestone 1: updated documentation for the new world split
    • Milestone 2: documentation for world-runtime, world-session, and the first corrected game slice
    • Milestone 3: documentation for dashboard composition surfaces
    • Milestone 4: documentation for repositories, sync, and legacy boundaries
    • Milestone 5: consolidation, cross-linking, and cleanup

    Technical closure criteria for a milestone¶

    A milestone is only considered closed when all the following are true:

    1. the game starts in development mode without new runtime errors
    2. the agreed manual smoke test passes completely
    3. there is no visible regression in:
    4. world loading
    5. real activity -> gold flow
    6. base persistence
    7. main dashboard behavior
    8. the target responsibility of the milestone no longer lives in the original source file/module
    9. the new boundary or responsibility split is documented
    10. all transitional wrappers, facades, or compatibility shims introduced in the milestone are inventoried
    11. frozen hotspot files did not grow except for strictly temporary extraction wiring
    12. typecheck/build complete without new errors
    13. there are no broken imports introduced by the milestone
    14. no new circular dependencies are detected in the affected graph

    Transitional wrapper policy¶

    Temporary wrappers/facades are allowed, but they must be controlled explicitly.

    Every transitional wrapper must record:

    • purpose
    • owner
    • provisional public contract
    • current consumers
    • target removal milestone or explicit review date
    • whether it exists for migration, compatibility, or sequencing reasons

    Temporary wrappers are not neutral. They are debt accepted in exchange for safer migration and must be reviewed as such.

    Early infrastructure exception¶

    Infrastructure may be advanced earlier only when it directly unlocks world-runtime or world-session.

    This exception does not authorize: - broad sync cleanup in parallel - opportunistic repository redesign for convenience - side refactors that do not reduce the critical path of the current milestone

    Consequences¶

    Positive consequences¶

    • risk is reduced earlier by clarifying the densest core first
    • the team gets value from each milestone without waiting for the whole program to finish
    • dashboard decomposition happens on top of clearer lower-level contracts
    • the refactor becomes easier to validate incrementally

    Trade-offs¶

    • some intermediate architecture may look uneven during transition
    • temporary facades may be necessary for safe sequencing
    • a world-first approach demands discipline so it does not become a permanent excuse to postpone other needed work

    Review policy¶

    This ADR defines the current execution strategy.

    Unlike ADR-0001, this strategy may be revised if technical evidence shows that a different sequence would reduce risk or improve delivery.

    Any revision should preserve:

    • milestone-level stability
    • world/runtime clarity as an early concern
    • documentation continuity
    • controlled use of transitional wrappers
