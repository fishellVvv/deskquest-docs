    ADR-0001 - DeskQuest adopts a modular layered monolith architecture¶

    • Status: Accepted
    • Date: 2026-03-27
    • Deciders: DeskQuest team
    • Scope: Product architecture and codebase organization
    • Tags: architecture, monolith, layering, modularity

    Context¶

    DeskQuest has grown from an early prototype into a product with multiple visible surfaces, a shared persistent world, native integrations, and increasing systemic complexity.

    The project already expresses several strong product constraints:

    • it is a desktop-first game/application, not a distributed platform
    • idle, dashboard, and game are distinct product surfaces, but they must remain connected through the same world
    • native integration, persistence, telemetry, and window coordination are core concerns
    • dimensions/themes must not take control over the core rules of the game
    • future midgame systems will increase the pressure on architecture boundaries

    The codebase review identified a recurring risk pattern:

    • product logic, runtime orchestration, persistence, and UI composition are not yet separated clearly enough
    • some high-density files and providers are accumulating responsibilities that belong to different layers
    • the project needs stronger architectural boundaries before additional systemic growth

    A full rewrite is not justified.

    Decision¶

    DeskQuest adopts a modular layered monolith architecture.

    This means:

    • DeskQuest remains a single application, built, versioned, and shipped as one product
    • the application is organized through explicit modules and responsibility boundaries
    • the architecture is layered so that each concern has a primary home
    • game rules remain centered in domain-oriented code
    • UI composition, runtime orchestration, and platform integration are made explicit instead of remaining hidden in large React entrypoints or providers

    Architectural model¶

    The target architecture is organized around these layers and areas:

    domain¶

    Holds the core game rules and product invariants.

    Examples: - world model and invariants - world commands and selectors - simulation logic - gameplay rules for active slices - progression rules and non-visual game systems

    domain must not depend on: - React - Tauri - window APIs used as integration boundaries - local persistence details - visual theming runtime

    application¶

    Coordinates use cases, sessions, runtime orchestration, and flows between layers.

    Examples: - world runtime/session lifecycle - orchestration of telemetry application into world updates - use-case level services for product flows - coordination between persistence and domain simulation - integration-level actions requested by visible features

    application may depend on: - domain - platform

    platform¶

    Encapsulates interaction with the real execution environment.

    Examples: - Tauri command adapters - storage repositories - BroadcastChannel and window event integration - native telemetry bridges - window management adapters - compatibility adapters for legacy runtime/storage contracts

    platform must not become a second domain layer. It exists to adapt environment concerns, not to define gameplay rules.

    features¶

    Contains visible user-facing experiences, organized by product surface or feature slice.

    Examples: - features/idle - features/dashboard - features/game

    features should contain: - React components - feature-level hooks - local composition logic - visual state handling related to presentation

    features should not become the primary home of game simulation rules.

    app¶

    Contains composition of the frontend application.

    Examples: - entry points - provider composition - shell composition - route/view assembly

    app is a composition layer, not a hidden runtime engine.

    themes¶

    Contains dimension/theme runtime, visual tokens, assets, and packaging concerns.

    Themes may shape presentation and dimension-specific content packaging, but they do not control the core rules of the game.

    shared¶

    Contains truly cross-cutting low-level utilities.

    This area must remain disciplined. It is not a fallback bucket for unclear ownership.

    Dependency principles¶

    The architecture follows these dependency expectations:

    • features may depend on application, domain, and presentation-safe shared utilities
    • application may depend on domain and platform
    • platform may depend on environment/tooling integrations, but not on React-driven presentation concerns
    • domain must remain independent from React and Tauri
    • themes may integrate with features and platform where needed for presentation/runtime, but not redefine domain rules

    Consequences¶

    Positive consequences¶

    • gameplay rules gain a more stable home
    • runtime orchestration becomes visible and reviewable
    • platform dependencies become easier to isolate and replace
    • active slices can grow without pushing simulation logic into UI folders
    • dashboard surfaces can be simplified into composition shells over time
    • architectural documentation becomes easier to keep aligned with code structure

    Trade-offs¶

    • some temporary duplication or wrappers may appear during migration
    • contributors must be more deliberate when choosing ownership for new code
    • some files may need transitional adapters to preserve stability while boundaries are extracted
    • architectural discipline will need to be enforced continuously during the refactor period

    Rejected alternatives¶

    Full rewrite¶

    Rejected because the current project already has enough valid structure and product value to justify iterative refactoring instead.

    Microservices or distributed decomposition¶

    Rejected because DeskQuest is not a distributed backend platform. Its primary problem is not deployment topology; it is responsibility clarity within a desktop product.

    Keep current structure and only split very large files¶

    Rejected because file size is a symptom, not the root issue. Splitting files without redefining ownership would only distribute the same ambiguity.

    Non-goals¶

    This ADR does not define:

    • the detailed order of refactor execution
    • milestone sequencing
    • closure criteria for each refactor phase
    • temporary migration wrappers or operational transition rules

    Those belong to execution strategy and are handled separately.

    Review policy¶

    This architectural decision is intended to be stable.

    The implementation strategy may evolve, but the following principles should not be relaxed without an explicit new ADR or ADR amendment:

    • DeskQuest remains a modular layered monolith
    • domain stays independent from React and Tauri
    • application is the orchestration layer
    • platform is the integration layer
    • features are presentation-oriented experience modules
    • themes do not govern core gameplay rules
