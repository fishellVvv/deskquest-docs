    Arquitectura actual¶

    Objetivo¶

    Esta pagina describe la forma viva del proyecto hoy.

    No cuenta el historial del refactor.

    Su funcion es permitir que cualquier persona del equipo entienda la estructura actual sin reconstruir M0 a M4b.

    Si aparece una discrepancia entre esta pagina y un ADR aceptado, el ADR manda.

    Forma general del proyecto¶

    DeskQuest funciona hoy como un monolito modular por capas.

    La experiencia visible sigue dividida en tres superficies de producto:

    • idle
    • dashboard
    • game

    Sobre esa experiencia, la arquitectura se organiza en estas capas:

    • domain
    • application
    • platform
    • features
    • themes

    1. domain¶

    Vive en src/domain.

    Aqui deben vivir:

    • modelo de juego
    • reglas puras
    • selectores
    • comandos puros
    • simulacion

    Ejemplo fuerte actual:

    • world-model.ts
    • world-selectors.ts
    • world-commands.ts
    • world-simulation.ts

    Tambien queda ya fijado un precedente importante para game:

    • logging-mission.ts

    domain no debe depender de React, de Tauri ni de detalles de una dimension concreta.

    2. application¶

    Vive en src/application.

    Aqui deben vivir:

    • casos de uso
    • runtime de aplicacion
    • sesion viva
    • orquestacion entre dominio y plataforma
    • pegamento que no pertenece ni a React ni a domain

    Ejemplo fuerte actual:

    • world-session.ts
    • world-runtime.ts
    • world-actions.ts
    • world-react-state.ts

    Esta capa ya existe para que WorldProvider no vuelva a absorber runtime u orquestacion de negocio.

    3. platform¶

    Vive en src/platform.

    Aqui deben vivir:

    • persistencia
    • sincronizacion
    • integracion con entorno real
    • backends de almacenamiento o canales

    Ejemplo fuerte actual:

    • world-snapshot-storage.ts
    • world-runtime-storage.ts

    platform no deberia decidir reglas del juego.

    Su trabajo es servir contratos claros a application.

    4. features¶

    Vive en src/features.

    Aqui deben vivir:

    • superficies visibles del producto
    • composicion de UI
    • hooks de coordinacion de feature
    • entrada visible a slices activos

    Ejemplos importantes:

    • DashboardCourtyard.tsx
    • DashboardSettingsRoom.tsx
    • useCourtyardLoggingMission.ts

    La regla importante es esta:

    • features presenta y coordina experiencia
    • no deberia volver a alojar simulacion jugable pura

    5. themes¶

    Vive en src/themes.

    Aqui deben vivir:

    • resolucion visual de dimensiones
    • assets
    • tokens
    • lexicon
    • registro runtime de dimensiones

    themes decide presentacion y lenguaje visual.

    No debe gobernar reglas del mundo, progreso o economia.

    Providers y shell¶

    Los providers siguen siendo importantes, pero ya no deben actuar como capa oculta de negocio.

    El caso mas importante es WorldProvider.tsx:

    • hoy es shell React sobre piezas de application/world
    • ya no gobierna el runtime del mundo por si mismo

    Ese es uno de los precedentes mas importantes que no deben revertirse.

    Precedentes estructurales que ya consideramos buenos¶

    world¶

    world ya es el ejemplo canonico de separacion entre:

    • modelo
    • lecturas
    • comandos
    • simulacion
    • acciones de aplicacion
    • persistencia/sync de plataforma

    Primer slice de game¶

    Dirigir tala ya deja un precedente valido:

    • logica pura en domain/game
    • orquestacion de feature fuera de la UI principal
    • pantalla visible dentro de features

    Dashboard¶

    El dashboard ya no debe volver a crecer como un unico archivo que mezcla:

    • composicion visual
    • seleccion local
    • overlays
    • wiring de slices activos
    • integraciones pesadas

    Lo que no debe volver a pasar¶

    • meter runtime de negocio dentro de providers
    • esconder persistencia/sync dentro de domain
    • hacer crecer dashboard mediante bloques grandes mezclados
    • meter simulacion de game directamente en features
    • dejar que themes dicte reglas del juego
    • mantener legacy por inercia si ya no protege nada vivo
