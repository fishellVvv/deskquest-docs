    Plan de migracion a DeskQuest¶

    Estado¶

    • completada
    • fecha de preparacion: 2026-03-24
    • progreso actual:
    • fases 1 a 7: completadas
    • siguiente paso oficial: generar y validar la primera build oficial 0.1.0-beta.1

    Objetivo¶

    Ejecutar el paso desde la identidad provisional test_game hacia la identidad aprobada DeskQuest sin romper:

    • builds empaquetadas
    • persistencia del usuario
    • soporte y QA
    • trazabilidad documental

    Este documento define el bloque tecnico del rename y su orden de cierre. El rename ya ha sido ejecutado y validado; la pagina queda como registro operativo de como se cerró la transicion desde test_game a DeskQuest.

    Fuente de verdad asociada¶

    • Identidad de aplicacion

    Identidad de origen y destino¶

    Capa Origen actual Destino aprobado
    nombre visible test_game DeskQuest
    repositorio test_game deskquest
    carpeta principal test_game deskquest
    ejecutable interno test_game.exe DeskQuest.exe
    save/config namespace test_game deskquest
    app id io.github.fishellvvv.testgame com.fishellvvv.deskquest

    Regla de ejecucion¶

    La migracion debe hacerse como un bloque tecnico entero.

    No conviene distribuir una build de testers entre medias con identidad mezclada, por ejemplo:

    • branding nuevo con guardados viejos sin migracion
    • app id nuevo con namespaces de localStorage viejos
    • repo renombrado pero documentacion aun atada a test_game

    Fase 1. Branding y metadatos visibles (completada)¶

    Actualizar:

    • README.md
    • docs/index.md
    • mkdocs.yml
    • site_name y referencias visibles al producto
    • copys visibles que aun usen test_game

    Objetivo:

    • que la documentacion y la UI hablen del producto como DeskQuest
    • sin asumir todavia que la migracion tecnica ya esta completada

    Fase 2. Repo y metadatos de build (completada)¶

    Actualizar:

    • nombre del repositorio a deskquest
    • carpeta principal local a deskquest
    • package.json
    • name
    • repository
    • bugs
    • homepage
    • src-tauri/Cargo.toml
    • package.name
    • repository
    • lib.name
    • src-tauri/src/main.rs
    • llamada al crate renombrado
    • src-tauri/tauri.conf.json
    • productName
    • identifier
    • app.windows[].title

    Objetivo:

    • que repo, bundle y ejecutable queden ya alineados con DeskQuest

    Fase 3. Namespace de guardado web (completada)¶

    Actualizar las claves nuevas del runtime:

    Sistema Clave actual Clave objetivo
    configuracion test_game.configuration deskquest.configuration
    telemetria test_game.telemetry deskquest.telemetry
    mundo test_game.world deskquest.world
    heartbeat runtime test_game.world.runtime_heartbeat deskquest.world.runtime_heartbeat
    dimension activa test_game.active_dimension deskquest.active_dimension
    evento de dimensiones test_game.installed_dimensions_changed deskquest.installed_dimensions_changed

    Compatibilidad minima obligatoria:

    • leer primero la clave nueva
    • si no existe, leer la clave vieja
    • si la vieja existe, migrarla y persistir inmediatamente en la nueva

    Compatibilidad legacy que debe mantenerse:

    • test_game.active_theme
    • test_game.theme_library_reload

    Esas claves legacy ya vienen de migraciones anteriores y no conviene perderlas al mismo tiempo que cambiamos el branding del proyecto.

    Fase 4. Perfil nativo y AppData (completada)¶

    Con el cambio de app id, las rutas del perfil cambian.

    Rutas actuales:

    • AppData\\Local\\io.github.fishellvvv.testgame
    • AppData\\Roaming\\io.github.fishellvvv.testgame

    Rutas objetivo:

    • AppData\\Local\\com.fishellvvv.deskquest
    • AppData\\Roaming\\com.fishellvvv.deskquest

    Datos sensibles que deben sobrevivir:

    • EBWebView
    • logs
    • telemetry\\global-activity.json
    • installed-dimensions
    • caches auxiliares del runtime

    Estrategia recomendada:

    1. en el arranque nativo, detectar si existe el perfil viejo y no existe aun el nuevo
    2. copiar o mover el contenido relevante al perfil nuevo
    3. dejar trazabilidad en logs
    4. no sobreescribir un perfil nuevo ya existente

    Regla:

    • si el perfil nuevo ya existe, no migrar a ciegas
    • si solo existe el viejo, migrar automaticamente

    Fase 5. Scripts y tooling (completada)¶

    Actualizar:

    • tools/reset-dev-progress.ps1
    • utilidades o aliases que apunten al app id viejo
    • documentacion de rutas de instalacion y datos del usuario

    Objetivo:

    • que el tooling de desarrollo no siga limpiando solo el perfil viejo
    • que soporte y QA sepan donde mirar tras el rename

    Fase 6. Documentacion y soporte (completada)¶

    Actualizar y resincronizar:

    • Instalacion y datos de usuario
    • Checklist QA beta
    • Validacion de build beta
    • Guia corta para testers
    • README
    • STATUS
    • versioning
    • worklog

    Nota importante:

    • tras la primera build renombrada, hay que revalidar la ruta real de instalacion en Windows en vez de asumirla por intuicion

    Fase 7. Validacion de salida (completada)¶

    La migracion no se considera cerrada hasta comprobar:

    1. build empaquetada arranca como DeskQuest
    2. el idle aparece correctamente
    3. el dashboard abre y no se duplica
    4. una sesion con datos antiguos migra y conserva progreso
    5. una sesion nueva crea datos solo bajo el namespace nuevo
    6. Dirigir tala sigue entregando y persistiendo bien
    7. reset y presets siguen funcionando
    8. la documentacion publica ya no presenta test_game como nombre del producto

    Paso inmediatamente posterior¶

    Con la migracion ya cerrada, el versionado profesional aceptado queda ya activado en:

    • Politica de versionado

    Estado actual:

    • la linea publica ya arranca en 0.1.0-beta.1
    • el siguiente bloque operativo es generar y validar la primera build oficial con esa version

    Riesgos concretos¶

    • que cambie el app id pero no el acceso al guardado, dejando a testers con progreso aparentemente perdido
    • que el perfil nuevo y el viejo convivan sin una politica clara
    • que el repo y la documentacion se renombren antes que el bundle
    • que el instalador use un nombre y la app otro
    • que QA y soporte hablen de DeskQuest mientras AppData sigue documentado como test_game

    Regla para el commit de migracion¶

    Conviene separar el trabajo asi:

    1. commit de plan y decision documental
    2. bloque tecnico de migracion real
    3. commit de validacion y documentacion final post-migracion

    Resultado esperado¶

    Al terminar:

    • el producto visible sera DeskQuest
    • los nuevos datos se guardaran bajo deskquest y com.fishellvvv.deskquest
    • los usuarios existentes conservaran progreso compatible
    • la documentacion y el soporte dejaran de hablar de test_game salvo en el historico o en notas de migracion
