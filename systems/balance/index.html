    Balance¶

    Esta seccion recoge la capa de balance editable del proyecto.

    Objetivo¶

    Evitar que costes, tiempos, conversiones y multiplicadores nazcan enterrados en constantes dispersas dentro del codigo.

    La idea base es:

    1. editar valores en tablas
    2. validar esos datos con una capa tipada
    3. consumirlos desde dominio y UI

    Estado actual¶

    La primera capa de balance ya existe en:

    • balance/economy.csv
    • balance/buildings.csv
    • balance/building-upgrades.csv
    • src/domain/balance/

    Que cubre ahora¶

    • conversion de pulsaciones a oro
    • coste y tiempo base de la fundacion del primer asentamiento
    • impacto inicial de poblacion sobre el tiempo de construccion
    • registro de las primeras construcciones tempranas
    • mejoras reutilizables por nivel para edificios como viviendas
    • costes mixtos de oro, madera y piedra para las obras del patio
    • tasas base de madera/min y piedra/min para las dos primeras ramas productivas
    • crecimiento temporal de poblacion cuando el aforo del feudo aumenta

    Regla de trabajo¶

    • las tablas son la fuente editable
    • la capa src/domain/balance/ es la fuente validada para la app
    • si aparece un nuevo sistema economico, debe evaluarse primero si merece una tabla nueva antes de introducir constantes ad hoc

    Que si y que no va a CSV¶

    Los .csv de balance/ son para datos estaticos y calibrables:

    • costes
    • tiempos
    • tasas base
    • multiplicadores
    • poblacion inicial
    • aforo extra de edificios como viviendas
    • niveles posteriores de un edificio ya construido

    No son la fuente de verdad del progreso vivo del reino.

    No deben moverse a CSV:

    • recursos acumulados
    • edificios ya construidos
    • timers en marcha
    • nombre del asentamiento
    • mano de obra asignada

    Esos datos pertenecen al WorldState persistente.

    Presets de desarrollo¶

    Para testear rapido sin sufrir toda la progresion, el proyecto usa presets JSON de desarrollo en:

    • dev-presets/world/

    Su seleccion activa se controla con:

    • dev-presets/active-world-preset.txt

    Comandos utiles:

    • npm run dev:preset:list
    • npm run dev:preset -- <preset>
    • npm run dev:preset:none

    Estos presets existen para pruebas y tuning rapido. No sustituyen ni a las tablas de balance ni al guardado normal del mundo.

    Niveles de edificio¶

    El proyecto separa ya dos capas:

    • buildings.csv define la construccion inicial de nivel 1
    • building-upgrades.csv define los niveles siguientes del mismo edificio

    Cada fila de building-upgrades.csv representa el estado resuelto del edificio una vez alcanzado ese nivel.

    Ejemplo practico:

    • Viviendas nivel 1 se construye desde buildings.csv
    • Viviendas nivel 2 y nivel 3 se calibran desde building-upgrades.csv

    Esto permite reutilizar la misma logica de dominio para:

    • costes
    • tiempo de obra
    • bonus del nivel alcanzado
    • futuras mejoras de edificios productivos

    Politica de precision y display¶

    Para no convertir el juego en una hoja de calculo, la precision interna y la presentacion visible no son exactamente la misma cosa.

    Regla actual:

    • los recursos acumulados (oro, madera, piedra, etc.) pueden guardarse con decimales en runtime
    • la UI principal muestra esos acumulados como enteros
    • las tasas o metricas de produccion (madera/min) pueden mostrarse con una sola cifra decimal
    • los costes de construccion se mantienen en enteros

    Motivo:

    • los decimales internos permiten que la produccion por tiempo transcurrido sea suave y facil de balancear
    • los enteros visibles mantienen la lectura limpia y menos "Excel"
    • separar ambas capas evita tener que elegir entre precision tecnica y claridad de interfaz
