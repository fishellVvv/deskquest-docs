    Especificacion tecnica de paquete de dimension externa¶

    Estado del contrato¶

    • version del esquema: 1
    • estado: contrato de referencia para comenzar la migracion del sistema actual de dimensiones compiladas a un sistema de paquetes externos cargables en runtime
    • dimension por defecto obligatoria: builtin:medieval

    Objetivo¶

    Definir un formato exacto para dimensiones externas que cumpla estas condiciones:

    • se puedan anadir, quitar y compartir sin recompilar la app
    • tengan fallback total a la dimension medieval integrada
    • permitan traducciones propias por dimension
    • no puedan ejecutar codigo arbitrario
    • preparen la migracion del idle a un sistema basado en assets y metadatos

    Decision de empaquetado¶

    Formatos soportados¶

    1. carpeta descomprimida para desarrollo local y documentacion
    2. archivo empaquetado .dimension
    3. archivo .zip compatible durante la fase de transicion y para soporte tecnico/manual import

    Formato recomendado para compartir¶

    .dimension

    La extension .dimension no deja de ser un zip por dentro, pero nos da varias ventajas:

    • lenguaje propio del proyecto
    • posibilidad de validar el tipo de archivo de forma mas clara
    • experiencia mas simple para usuarios y modders
    • evita depender de rar

    Formato no objetivo¶

    .rar

    No se recomienda como formato oficial porque complica el soporte tecnico sin aportar una ventaja real frente a zip/.dimension.

    Modelo de ubicacion¶

    Dimension integrada¶

    • vive dentro de la app
    • no se distribuye como paquete externo
    • id reservado: builtin:medieval
    • actua siempre como fallback de ultima instancia

    Paquetes importables¶

    Los usuarios comparten dimensiones como archivos .dimension.

    La app no ejecuta esos archivos directamente desde una carpeta escaneable, sino que los importa uno a uno desde configuracion y los instala descomprimidos en una carpeta interna de dimensiones instaladas.

    Dimensiones instaladas¶

    Una vez importada, la dimension pasa a vivir en una ruta interna parecida a esta:

    <app-data>/
  installed-dimensions/
    io.github.autor.dimension-ejemplo/
      manifest.json
      tokens.json
      locales/
      assets/

    Regla de producto¶

    • builtin:medieval siempre existe
    • Base lunar se distribuye como .dimension oficial de ejemplo
    • el resto de dimensiones opcionales siguen el mismo flujo
    • la beta actual centra la UX visible en importar y activar

    Reglas de identificadores¶

    Namespace reservado¶

    Los ids que empiecen por builtin: quedan reservados al nucleo.

    Recomendacion¶

    Usar ids estables de estilo dominio invertido o namespace personal:

    • com.usuario.nombre-dimension
    • io.github.autor.pagoda-amber

    Ejemplos validos¶

    • io.github.autor.bastion-demo
    • io.github.autor.moon-temple

    Ejemplos no validos¶

    • builtin:castle-variant
    • Medieval
    • dimension rara 01

    Estructura exacta del paquete¶

    Tanto la carpeta descomprimida como el archivo .dimension deben contener esta estructura logica:

    my-dimension.dimension
  manifest.json
  tokens.json
  locales/
    es.json
    en.json
  assets/
    idle/
      structure.webp
      layout.json
    ui/
      motif.svg
      marker.webp
  fonts/
    display.woff2
    body.woff2
  preview.png

    Archivo manifest.json¶

    Archivo obligatorio. Define identidad, herencia y rutas internas del paquete.

    Ejemplo completo¶

    {
  "schemaVersion": 1,
  "id": "io.github.autor.bastion-demo",
  "label": "Bastion de cobre",
  "description": "Un bastion industrial de cobre, humo y remaches.",
  "version": "1.0.0",
  "author": "DeskQuest Team",
  "homepage": "https://ejemplo.dev",
  "defaultLocale": "es",
  "extends": "builtin:medieval",
  "uiHints": {
    "panelStyle": "stone"
  },
  "files": {
    "tokens": "tokens.json",
    "localesDir": "locales",
    "idleLayout": "assets/idle/layout.json",
    "idleStructure": "assets/idle/structure.webp",
    "accentMotif": "assets/ui/motif.svg",
    "notificationMarker": "assets/ui/marker.webp",
    "preview": "preview.png",
    "fontDisplay": "fonts/display.woff2",
    "fontBody": "fonts/body.woff2"
  }
}

    Campos obligatorios¶

    • schemaVersion: number
    • id: string
    • label: string
    • description: string
    • version: string
    • defaultLocale: string
    • files.tokens: string
    • files.localesDir: string
    • files.idleLayout: string

    Campos opcionales¶

    • author: string
    • homepage: string
    • extends: string
    • uiHints.panelStyle: "stone" | "glass"
    • files.idleStructure: string
    • files.accentMotif: string
    • files.notificationMarker: string
    • files.preview: string
    • files.fontDisplay: string
    • files.fontBody: string

    Reglas duras¶

    • schemaVersion debe coincidir con la version soportada por la app
    • id no puede usar el prefijo builtin:
    • defaultLocale debe existir realmente en locales/
    • extends, si existe, debe apuntar a una dimension conocida o a builtin:medieval
    • todas las rutas de files deben ser relativas al paquete
    • no se permiten rutas absolutas ni secuencias tipo ../

    Archivo tokens.json¶

    Contiene solo tokens visuales conocidos por el sistema. Puede ser parcial.

    Claves validas¶

    • fontDisplay
    • fontBody
    • neutral100 ... neutral900
    • primary100 ... primary900
    • accent100 ... accent900
    • danger100 ... danger900
    • focusRing
    • overlay
    • glow
    • shadowPanel

    La app deriva desde esta paleta las variables semanticas internas (bg, surface, text, border, etc.), de modo que una dimension futura pueda recolorear toda la UI tocando solo tokens.json.

    Regla¶

    Si falta una clave:

    1. se busca en la dimension activa
    2. despues en la dimension padre indicada en extends
    3. despues en builtin:medieval
    4. si sigue faltando, se usa un fallback hardcoded del sistema

    Carpeta locales/¶

    Cada archivo locales/<lang>.json contiene una traduccion parcial de los textos tematicos de la dimension.

    Claves validas actuales¶

    • primaryStructureLabel
    • dashboardHubLabel
    • dashboardHubTitle
    • dashboardHubDescription
    • dashboardLabel
    • dashboardPreviewLabel
    • settingsRoomLabel
    • settingsRoomTitle
    • settingsRoomDescription
    • returnToHubLabel
    • interactionCountLabel
    • interactionCountDetailLabel
    • dimensionSettingLabel
    • dimensionSettingDetailLabel
    • dimensionSelectorLabel
    • travelDimensionLabel
    • attentionLabel
    • readyLabel
    • openDashboardLabel
    • closeDashboardLabel
    • quitApplicationLabel
    • flavorLine

    Ejemplo¶

    {
  "primaryStructureLabel": "Bastion de cobre",
  "dashboardHubLabel": "Patio de maquinas",
  "dashboardHubTitle": "Patio de maquinas",
  "dashboardHubDescription": "Desde aqui se distribuyen las ordenes internas.",

  "dashboardPreviewLabel": "Resumen del bastion",
  "settingsRoomLabel": "Taller de ajuste",
  "settingsRoomTitle": "Taller de ajuste",
  "settingsRoomDescription": "Parametros y disposiciones del bastion.",
  "returnToHubLabel": "Volver al patio de maquinas",
  "interactionCountLabel": "Pulsaciones",
  "interactionCountDetailLabel": "Clicks y teclas acumulados",
  "dimensionSettingLabel": "Dimension activa",
  "dimensionSettingDetailLabel": "Cambia la dimension visible sin alterar la logica interna.",
  "dimensionSelectorLabel": "Dimension de destino",

    Regla de resolucion de idioma de dimension¶

    Para una clave tematica se seguira esta cadena:

    1. idioma seleccionado en la dimension activa
    2. defaultLocale de la dimension activa
    3. idioma seleccionado en la dimension padre
    4. defaultLocale de la dimension padre
    5. idioma seleccionado en builtin:medieval
    6. defaultLocale de builtin:medieval
    7. fallback hardcoded del sistema

    Archivo assets/idle/layout.json¶

    Archivo obligatorio. Define metadatos del idle para que el render futuro no dependa de ramas hardcodeadas por dimension.

    Sistema de coordenadas¶

    • origen: esquina superior izquierda del canvas del arte
    • unidad: pixel del arte fuente
    • los valores se interpretan antes de cualquier escalado del idle

    Ejemplo¶

    {
  "canvasWidth": 300,
  "canvasHeight": 128,
  "artworkLabel": "Fortaleza pixelada",
  "dragSafeInset": {
    "right": 116,
    "bottom": 28
  },
  "gateHitbox": {
    "x": 144,
    "y": 78,
    "width": 50,
    "height": 44
  },
  "gateFeedback": {
    "x": 150,
    "y": 82,
    "width": 34,
    "height": 40
  },
  "atmosphere": {
    "x": 196,
    "y": 18,
    "width": 104,
    "height": 40
  }
}

    Campos obligatorios¶

    • canvasWidth: number
    • canvasHeight: number
    • dragSafeInset.right: number
    • dragSafeInset.bottom: number
    • gateHitbox.x: number
    • gateHitbox.y: number
    • gateHitbox.width: number
    • gateHitbox.height: number
    • gateFeedback.x: number
    • gateFeedback.y: number
    • gateFeedback.width: number
    • gateFeedback.height: number
    • atmosphere.x: number
    • atmosphere.y: number
    • atmosphere.width: number
    • atmosphere.height: number

    Campo opcional¶

    • artworkLabel: string

    Funcion de cada bloque¶

    • canvasWidth / canvasHeight: tamano base del arte fuente
    • dragSafeInset: reserva minima para no inutilizar la interaccion principal al arrastrar
    • gateHitbox: zona clicable que abre el dashboard
    • gateFeedback: region donde se dibuja el brillo/feedback de puerta
    • atmosphere: region para halo, humo, luna, antorcha u otro elemento decorativo de atencion

    Assets soportados¶

    Tipos permitidos en v1¶

    • imagenes raster: .png, .webp, .jpg, .jpeg
    • imagenes vectoriales: .svg
    • fuentes: .woff2, .woff, .ttf

    Assets visuales conocidos¶

    • idleStructure
    • accentMotif
    • notificationMarker
    • preview
    • fontDisplay
    • fontBody

    Si faltan, se heredan de extends o de builtin:medieval.

    Traducciones: separacion obligatoria entre core y dimension¶

    core locales¶

    Textos generales de la app:

    • configuracion
    • danger zone
    • errores
    • validaciones
    • botones comunes

    Estos textos no viven dentro del paquete de dimension en v1.

    dimension locales¶

    Textos diegeticos y tematicos:

    • nombre del edificio principal
    • nombre de salas
    • flavor text
    • titulos narrativos de la UI

    Estos si viven dentro del paquete de dimension.

    Consecuencia¶

    Una dimension puede aportar idiomas que la app base todavia no tenga. En ese caso se aceptara una mezcla controlada: UI core en un idioma base y textos tematicos en el idioma propio aportado por esa dimension.

    Fallback exacto¶

    Tokens¶

    1. dimension activa
    2. padre (extends)
    3. builtin:medieval
    4. fallback hardcoded

    Assets¶

    1. dimension activa
    2. padre (extends)
    3. builtin:medieval
    4. asset nulo permitido solo si el sistema sabe continuar sin el

    Textos de dimension¶

    1. dimension activa en idioma seleccionado
    2. dimension activa en defaultLocale
    3. padre en idioma seleccionado
    4. padre en defaultLocale
    5. builtin:medieval en idioma seleccionado
    6. builtin:medieval en su defaultLocale
    7. fallback hardcoded

    Restricciones de seguridad¶

    Los paquetes de dimension no pueden ejecutar codigo.

    Permitido¶

    • json
    • imagenes
    • fuentes
    • audio mas adelante

    Prohibido¶

    • js
    • ts
    • css arbitrario
    • html
    • binarios ejecutables

    Validacion minima para aceptar una dimension¶

    Una dimension se rechazara si ocurre cualquiera de estas condiciones:

    • falta manifest.json
    • schemaVersion no es compatible
    • falta defaultLocale
    • faltan files.tokens, files.localesDir o files.idleLayout
    • el id usa builtin:*
    • falta el archivo locales/<defaultLocale>.json
    • layout.json no define gateHitbox
    • alguna ruta intenta salir del paquete
    • el paquete contiene tipos de archivo prohibidos en ubicaciones conocidas

    Flujo recomendado para modders¶

    1. duplicar examples/skins/io.github.fishellvvv.lunar-base
    2. cambiar manifest.json, tokens.json, locales y assets
    3. comprimir la carpeta como .dimension
    4. abrir la app
    5. ir a configuracion
    6. pulsar Anadir dimension...
    7. elegir el archivo .dimension
    8. si ya existe una con el mismo id, decidir si se sobrescribe
    9. comprobar que aparece en Instaladas
    10. pulsar su boton para activarla

    Regla de conflictos¶

    • si coincide manifest.id, la app trata el paquete como la misma dimension y pide confirmacion para sobrescribir
    • si el id es distinto pero el nombre visible coincide, la app avisa y permite continuar

    Nota sobre eliminacion¶

    • builtin:medieval no se puede quitar
    • la beta actual no expone eliminacion desde la UI de configuracion
    • el fallback a builtin:medieval sigue siendo la garantia base del runtime

    Demo oficial de referencia¶

    La primera dimension de ejemplo mantenida por el proyecto es:

    • examples/skins/io.github.fishellvvv.lunar-base

    Esta carpeta debe servir como:

    • ejemplo de paquete valido
    • base para construir lunar-base.dimension
    • referencia minima para modders

    Su README.md interno explica que controla cada archivo, como duplicar el paquete y como importarlo en la app.
