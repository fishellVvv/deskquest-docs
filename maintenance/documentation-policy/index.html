    Politica de mantenimiento documental¶

    Esta politica fija las reglas practicas para que la documentacion del proyecto crezca de forma ordenada y siga siendo util.

    1. Principio general¶

    La documentacion de DeskQuest se divide en dos capas:

    • documentacion narrativa: explica producto, arquitectura, onboarding, sistemas y decisiones
    • documentacion de referencia: expone tipos, hooks y modulos del runtime a partir del propio codigo

    La narrativa vive en docs/ y se publica a traves de MkDocs. La referencia tecnica se genera desde TypeDoc y no se versiona.

    2. Fuente de verdad por tipo de documento¶

    README¶

    README.md debe ser corto.

    Solo debe contener:

    • que es el proyecto
    • su estado actual a alto nivel
    • comandos principales de desarrollo
    • enlace al portal documental

    No debe convertirse en un duplicado del roadmap ni del worklog.

    STATUS¶

    STATUS.md vive en la raiz como foto viva del proyecto.

    Su funcion es dar una vista rapida y siempre consultable de:

    • version interna
    • fase activa
    • ultima accion cerrada
    • direccion inmediata recomendada
    • estado de las lineas principales del proyecto

    Debe mantenerse breve, orientado a lectura diaria y sincronizado con el worklog, no sustituirlo.

    Portal docs/¶

    El portal es la fuente de verdad de la documentacion humana.

    Distribucion actual:

    • docs/getting-started/: onboarding, mapa del repo y ruta de lectura
    • docs/product/: roadmap, versioning y worklog
    • docs/product/deferred-improvements.md: backlog de mejoras no bloqueantes
    • docs/architecture/: estructura y lenguaje interno del proyecto
    • docs/api/: guia de la referencia tecnica generada
    • docs/systems/: documentacion de sistemas concretos
    • docs/maintenance/: reglas de mantenimiento documental

    Worklog¶

    docs/product/worklog.md registra bloques reales de trabajo.

    Debe actualizarse cuando se cierre un avance relevante:

    • refactorizacion
    • nueva funcionalidad
    • migracion
    • limpieza importante
    • fase documental cerrada

    No hace falta crear una entrada por microcambio trivial.

    Versioning¶

    docs/product/versioning.md refleja tres capas distintas:

    • version publica activa del producto
    • hito interno ligado al worklog
    • version del guardado

    Regla practica:

    • el versionado publico sigue SemVer con prereleases
    • el hito interno sigue ligado a entradas relevantes del worklog
    • la version del guardado solo cambia si cambia el formato persistido

    Roadmap¶

    docs/product/roadmap.md refleja direccion y prioridades.

    Debe actualizarse cuando cambien:

    • objetivos de producto
    • fases activas
    • backlog prioritario
    • decisiones que alteren el orden del desarrollo

    Pendientes diferidos¶

    docs/product/deferred-improvements.md centraliza mejoras y deuda tecnica no bloqueante.

    Debe actualizarse cuando:

    • aparece una mejora real que no conviene abordar en el foco actual
    • se decide posponer un pulido para no romper el ritmo del producto
    • una mejora diferida deja de tener sentido o pasa a ser prioritaria

    3. Que merece TSDoc¶

    No todo el codigo necesita comentarios de referencia.

    Se documentan con TSDoc:

    • providers
    • stores o capas de persistencia
    • runtime de dimensiones
    • clientes Tauri
    • hooks con logica no trivial
    • contratos tecnicos y tipos nucleares

    Normalmente no merece TSDoc extensa:

    • componentes de UI obvios
    • helpers pequenos y evidentes
    • estilos o wiring trivial

    La regla es simple:

    • si alguien nuevo necesita entender el proposito del modulo antes de leerlo, conviene documentarlo

    4. Superficie de API publica¶

    La referencia TypeScript no se genera sobre todo src/.

    La superficie publica se controla desde:

    • src/reference/typescript-api.ts

    Solo deben exportarse ahi los modulos que:

    • representen contratos estables
    • tengan valor arquitectonico
    • sirvan para onboarding o consulta tecnica

    Si un modulo no debe convertirse en referencia publica, no se exporta desde ese punto de entrada.

    5. Archivos generados¶

    La salida generada no se versiona.

    Esto incluye:

    • site/
    • site/reference/typescript/

    Lo versionable es:

    • mkdocs.yml
    • typedoc.json
    • docs/
    • src/reference/typescript-api.ts
    • comentarios TSDoc del codigo

    6. Cuando actualizar la documentacion¶

    Debe actualizarse documentacion en el mismo bloque de trabajo si ocurre al menos uno de estos casos:

    • cambia el comportamiento visible del producto
    • cambia la arquitectura o la nomenclatura
    • cambia el flujo para modders
    • cambia una API que ya forma parte de la referencia
    • se cierra una fase del roadmap

    7. Checklist minimo antes de cerrar un bloque¶

    Antes de dar por cerrado un bloque importante, revisar:

    1. README sigue siendo correcto
    2. STATUS.md sigue reflejando el estado real del proyecto
    3. worklog recoge el avance
    4. versioning refleja version publica, hito interno y version del guardado
    5. roadmap no ha quedado desfasado
    6. si se tocaron modulos nucleares, su TSDoc sigue siendo valida
    7. mkdocs build sigue funcionando
    8. si se toco la superficie API, npm run docs:api:ts sigue generando limpio

    8. Politica de decisiones futuras¶

    Si una decision arquitectonica futura cambia de forma fuerte el proyecto, hay dos niveles:

    • si basta con el roadmap y el worklog, no hace falta mas
    • si la decision tiene impacto duradero y varias alternativas reales, conviene crear un documento dedicado dentro de docs/architecture/

    No hace falta forzar ADR formales mientras el proyecto siga pequeno, pero la regla queda abierta para cuando aparezcan decisiones mas profundas.

    9. Resultado esperado¶

    Si seguimos esta politica:

    • la raiz del repo se mantiene limpia
    • el portal sigue siendo la entrada principal
    • la referencia tecnica no se descontrola
    • el conocimiento del proyecto sigue siendo consultable y escalable
