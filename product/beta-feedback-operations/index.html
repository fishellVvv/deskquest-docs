    Operativa de feedback beta¶

    Esta pagina convierte la fase actual de beta compartible en un flujo corto y repetible.

    DeskQuest 0.1.0-beta.2 ya no necesita mas trabajo de release para poder salir a colegas; lo que falta es transformar las primeras senales externas en una decision de producto clara.

    Objetivo inmediato¶

    1. compartir la build 0.1.0-beta.2 con colegas o testers cercanos
    2. centralizar la senal en el tracker publico de deskquest-docs
    3. separar rapido bloqueantes, fricciones repetidas y peticiones de slice nuevo
    4. decidir despues si conviene congelar la beta, abrir un microbloque correctivo o pasar al siguiente slice funcional

    Fuente de verdad del feedback¶

    Canal preferido:

    • Issues de deskquest-docs: https://github.com/fishellVvv/deskquest-docs/issues

    Reglas practicas:

    • usar Bug report para crashes, instalaciones rotas, perdida de progreso, bloqueos de onboarding o fallos que impidan jugar
    • usar Beta feedback para claridad, pacing, UX, impresion general y polish no bloqueante
    • si una senal llega por chat, oral o nota suelta, normalizarla despues en una issue o en una nota interna enlazable
    • no dejar decisiones de producto apoyadas solo en memoria oral o mensajes dispersos

    Que queremos aprender primero¶

    Antes de abrir features nuevas, la primera ronda debe responder estas preguntas:

    • se entiende el primer arranque sin explicacion adicional
    • el dashboard y el pergamino dejan clara la siguiente accion util
    • el pacing temprano de oro -> fundacion -> maderero / viviendas / poblacion se siente comprensible y no se rompe
    • Dirigir tala resulta legible, cerrable y coherente al volver al reino
    • la instalacion, la documentacion publica y el flujo de dimensiones no meten friccion innecesaria

    Clasificacion corta del feedback¶

    Clase Como reconocerla Destino recomendado
    bloqueante beta crash, perdida de guardado, instalacion fallida, ventana invisible, progreso imposible corregir antes de ampliar la ronda
    friccion fuerte el mismo roce aparece en 2 o mas testers, o rompe la lectura de los primeros 10-15 minutos valorar un microbloque correctivo
    senal de producto pide mas contenido, mas profundidad o una nueva actividad, pero la base actual aguanta usarla para elegir el siguiente slice, no para improvisar fixes grandes
    diferido preferencia aislada, polish menor o idea sin retorno inmediato mover a backlog diferido

    Senal minima para cerrar la primera lectura¶

    Se considera que ya hay material suficiente para una primera lectura cuando se cumple al menos una de estas condiciones:

    • aparece un bug claramente bloqueante
    • se acumulan 5 o mas issues/comentarios utiles
    • el mismo problema o confusion aparece en 2 testers distintos

    Regla de lectura inicial¶

    1. agrupar la senal por area: instalacion, onboarding, patio, Dirigir tala, Dimensiones, Configuracion y runtime
    2. distinguir si cada punto nace de un bug real, de copy/explicacion insuficiente o de falta de contenido
    3. redactar una conclusion corta y unica para la ronda, sin mezclar simultaneamente fixes grandes y features nuevas

    Salidas posibles de la primera ronda¶

    • congelar beta: si no aparecen bloqueantes y la mayoria del feedback es polish ligero o confirmacion de que la base aguanta
    • abrir microbloque correctivo: si aparecen 1-3 fricciones repetidas con retorno claro antes de ampliar testers
    • abrir siguiente slice funcional: si la base se sostiene y la senal pide claramente mas contenido o profundidad

    Candidatos si toca volver al desarrollo funcional¶

    Las lineas ya documentadas como candidatas naturales son:

    • Cantera jugable
    • mejoras activas de Dirigir tala
    • integracion del lore
    • nuevas zonas o slices de game

    Donde registrar el resultado¶

    • worklog: cierre del bloque de lectura de feedback
    • STATUS.md: foco actual y siguiente accion recomendada
    • roadmap: si cambia la prioridad real del producto
    • deferred-improvements: si aparecen ideas utiles pero no inmediatas
    • versioning: solo si cambia el hito interno o se prepara otra release
