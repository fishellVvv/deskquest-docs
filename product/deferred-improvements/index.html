    Pendientes diferidos¶

    Este documento centraliza mejoras, pulidos y deuda tecnica no bloqueante que han ido apareciendo a lo largo del proyecto.

    No son tareas olvidadas ni errores abiertos criticos: son frentes reales que conviene conservar visibles para retomarlos en el momento adecuado sin perder el hilo del producto.

    Como usar esta lista¶

    • Si un punto pasa a ser foco activo, debe moverse al roadmap o quedar reflejado en STATUS.md.
    • Si una mejora deja de tener sentido, debe retirarse de aqui al cerrar el bloque correspondiente en worklog.
    • Esta lista debe mantenerse agrupada por sistema, no por fecha.

    1. Captura global y telemetria¶

    Prioridad media¶

    • Sustituir el polling de snapshots por un modelo mas reactivo o por eventos nativos batched si el coste o la latencia acaban notandose.
    • Anadir un flush extra del contador nativo al cerrar la app para reforzar la persistencia en salidas bruscas.
    • Seguir vigilando el comportamiento de los hooks globales en contextos especiales de Windows y reevaluar Raw Input solo si aparecen limites reales.
    • Ampliar las pruebas manuales de escritorio a sesiones largas y a escenarios menos normales para detectar bordes de rendimiento o fiabilidad.

    Estado actual¶

    • La captura global ya es funcional y estable para el uso normal del producto.
    • El conteo interno y externo ya estan separados correctamente.

    2. Idle, arte y feedback de escritorio¶

    Prioridad media¶

    • Sustituir el pixelart placeholder del idle por arte definitivo.
    • Redisenar el feedback luminoso actual del idle cuando exista el arte final.
    • Definir mejor la metafora visual entre idle y dashboard para que la transicion se sienta mas propia del mundo y menos de prototipo.
    • Retomar el pulido del castillo/base solo cuando vuelva a tocar un bloque visual real y no antes.
    • Explorar audio ambiental y microanimaciones solo cuando la identidad visual este mas cerrada.

    Estado actual¶

    • El idle ya es estable, configurable y suficientemente claro como base tecnica.

    3. Dashboard y configuracion¶

    Prioridad media¶

    • Conectar de verdad las opciones de configuracion que hoy solo estan presentadas:
    • volumen
    • arranque con Windows
    • transparencia
    • animaciones / reduccion de movimiento
    • notificaciones
    • accesibilidad
    • atajos
    • comportamiento al cerrar
    • Completar la cobertura real de idioma a medida que crezca el dashboard.
    • Revisar periodicamente el copy del panel para que siga alineado con el lore de cada dimension.

    Estado actual¶

    • La base de configuracion ya es persistente, sincronizada entre ventanas y preparada para activar controles uno a uno.

    4. Dominio, progreso y gameplay¶

    Prioridad alta¶

    • Seguir encadenando la economia temprana mas alla de oro y madera.
    • Introducir crecimiento real de poblacion y su siguiente circuito economico.
    • Modelar progreso, recursos y misiones sobre una capa de dominio comun.
    • Construir el primer vertical slice jugable del modo game.
    • Conectar de forma visible idle, dashboard y game para que el mundo se sienta uno solo.

    Estado actual¶

    • La economia temprana ya ha arrancado y la siguiente frontera activa es madera -> viviendas -> crecimiento de poblacion.

    5. Dimensiones y modding¶

    Prioridad media¶

    • Cerrar la documentacion del flujo futuro para un repositorio online de dimensiones.
    • Decidir, mas adelante, si hace falta exponer una capa mas amigable de instalacion/actualizacion desde un indice remoto.
    • Mantener lunar-base.dimension como demo oficial viva de referencia para modders a medida que cambie el contrato.
    • Revisar de vez en cuando los pocos nombres internos legacy que quedan en la capa visual y nativa si empiezan a estorbar.

    Estado actual¶

    • El sistema de dimensiones ya esta estable para importacion local y modding manual.

    6. Rendimiento y runtime¶

    Prioridad media¶

    • Medir CPU y RAM reales del idle con mas sistematica.
    • Revisar suspension de tareas cuando haya ventanas ocultas o sin uso.
    • Asegurar que el futuro runtime del game no contamine el arranque del idle.
    • Mantener la regla de no ejecutar trabajo innecesario en segundo plano.

    Estado actual¶

    • El proyecto esta bien encaminado, pero todavia no ha entrado en la fase dura de optimizacion.

    7. Documentacion y tooling¶

    Prioridad baja¶

    • Anadir referencia Rust si el backend nativo sigue creciendo y empieza a merecer un portal equivalente al de TypeScript.
    • Empezar a usar documentos de decision mas formales en docs/architecture/ cuando aparezcan elecciones de sistema mas profundas.
    • Mantener esta lista viva para que los flecos no acaben otra vez repartidos entre roadmap, status y worklog.

    Estado actual¶

    • La base documental ya esta ordenada y estable.

    8. Regla de lectura¶

    Si una tarea de esta lista:

    • bloquea funcionalidad actual
    • rompe una promesa importante del producto
    • o pasa a ser foco inmediato

    entonces deja de ser un pendiente diferido y debe subir al roadmap activo y a STATUS.md.
