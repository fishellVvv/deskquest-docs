    Operativa de cierre de beta¶

    Esta pagina fija el orden de ejecucion recomendado para cerrar la beta de DeskQuest antes de volver a abrir desarrollo funcional grande.

    El objetivo es no perder pasos entre:

    • versionado publico
    • build compartible
    • limpieza de perfiles legacy
    • preparacion del repo
    • publicacion de documentacion
    • circuito de testers

    Principio general¶

    Mientras esta operativa no quede cerrada:

    • no conviene abrir nuevas features grandes
    • no conviene empezar Cantera jugable ni sistemas nuevos
    • solo deberian entrar fixes, tooling de release y pasos de soporte/publicacion

    Orden de ejecucion recomendado¶

    1. Activar el versionado profesional (completado)¶

    Separar de forma efectiva:

    • version publica del producto
    • hito interno de trabajo
    • version del guardado

    Decisiones ya aprobadas:

    • primera version publica: 0.1.0-beta.1
    • hito interno: el que corresponda en STATUS / worklog
    • save schema version: 1

    Archivos a alinear:

    • package.json
    • src-tauri/Cargo.toml
    • src-tauri/tauri.conf.json
    • README.md
    • Politica de versionado
    • docs de build y testers

    2. Generar y validar la primera build oficial 0.1.0-beta.1 (completado)¶

    Construir de nuevo:

    • .msi
    • setup.exe

    Y comprobar en la build real:

    • arranque del idle
    • apertura del dashboard
    • persistencia
    • Dirigir tala
    • cierre y reapertura
    • coherencia entre nombre visible, instalador y rutas de soporte

    3. Validar migracion de datos y hacer backup (completado)¶

    Antes de limpiar nada legacy:

    • confirmar que el perfil nuevo funciona bien
    • hacer backup manual de:
    • AppData\Local\com.fishellvvv.deskquest
    • AppData\Roaming\com.fishellvvv.deskquest
    • conservar temporalmente copia del perfil antiguo si aun existe

    Objetivo:

    • no perder la red de seguridad del rename

    4. Limpiar carpetas legacy fuera del repo (completado)¶

    Solo despues del backup y de validar la build nueva.

    Revisar y, si procede, eliminar:

    • AppData\Local\io.github.fishellvvv.testgame
    • AppData\Roaming\io.github.fishellvvv.testgame
    • instalaciones antiguas de test_game si aun existen

    Conviene documentar el procedimiento para soporte y para no volver a dudar en el futuro.

    5. Cerrar la estrategia de distribucion (completado)¶

    Decidir que se publicara y donde.

    Recomendacion actual:

    • usar GitHub Releases
    • no usar GitHub Packages por ahora

    Motivo:

    • Releases encaja muy bien con binarios de un juego de escritorio
    • Packages no aporta valor claro en esta fase

    6. Preparar la release beta en GitHub (completado)¶

    Crear:

    • tag v0.1.0-beta.1
    • release DeskQuest 0.1.0-beta.1

    Adjuntar:

    • .msi
    • setup.exe

    Preparar notas de release con:

    • alcance de la beta
    • que probar
    • limitaciones conocidas
    • donde reportar feedback

    Estado actual:

    • release viva: https://github.com/fishellVvv/deskquest/releases/tag/v0.1.0-beta.1

    7. Dejar el repo presentable (completado)¶

    Revisar:

    • About
    • descripcion
    • website
    • topics
    • social preview si interesa

    Topics sugeridos:

    • desktop-game
    • tauri
    • react
    • typescript
    • idle-game
    • incremental-game
    • medieval
    • pixel-art

    Tambien conviene revisar:

    • rama por defecto
    • proteccion de rama si aplica
    • visibilidad correcta del repo principal

    Estado actual:

    • repo principal ya tiene About, website y topics
    • website ya apunta al portal publico

    8. Crear el GitHub Project operativo (completado)¶

    Crear un unico Project, ligero, orientado a ejecucion.

    Campos recomendados:

    • Status
    • Type
    • Area
    • Priority
    • Beta blocker

    Columnas recomendadas:

    • Backlog
    • Ready
    • In progress
    • Blocked
    • Done

    Regla:

    • docs siguen siendo la fuente de verdad de producto
    • Project sirve para ejecutar trabajo, no para reemplazar arquitectura

    Estado actual:

    • proyecto operativo activo: DeskQuest Beta
    • columnas vivas: Backlog, Ready, In progress, Blocked, Done

    9. Preparar la publicacion publica de la documentacion (completado)¶

    Decidir el flujo:

    • repo principal privado
    • repo publico separado solo para docs

    La fuente de verdad de los .md sigue estando en este repo. Lo publico sera el resultado del portal generado.

    10. Publicar el portal documental (completado)¶

    Pasos previstos:

    • crear repo publico de docs
    • subir el portal generado o su flujo de publicacion
    • habilitar GitHub Pages
    • conseguir URL compartible

    Validar:

    • que la documentacion cargue bien
    • que no filtre nada sensible del repo privado

    Estado actual:

    • repo publico: deskquest-docs
    • GitHub Pages activo
    • URL viva: https://fishellvvv.github.io/deskquest-docs/

    11. Conectar repo principal y docs publicas (completado)¶

    Alinear enlaces entre:

    • repo principal
    • portal publico
    • guia de testers
    • docs de instalacion

    Objetivo:

    • que cualquier tester o colaborador sepa donde esta cada cosa sin preguntar

    Estado actual:

    • el repo principal ya enlaza al portal publico
    • la guia de testers y la docs de instalacion ya apuntan a la capa publica

    12. Preparar el circuito de testers (completado)¶

    Definir el paquete minimo de beta:

    • build
    • enlace a docs
    • guia corta para testers
    • forma de reportar bugs

    Tambien conviene decidir:

    • a quien se comparte primero
    • si se centraliza feedback en Issues
    • si se usa un formulario o una plantilla

    Estado actual:

    • feedback centralizado en Issues del repo publico de docs
    • plantillas activas de Bug report y Beta feedback
    • tracker publico: https://github.com/fishellVvv/deskquest-docs/issues

    13. Montar higiene minima del repo (completado)¶

    Muy recomendable antes de abrir mas el proyecto:

    • labels de issues
    • plantilla de bug
    • plantilla de feedback beta
    • CHANGELOG.md si compensa
    • CONTRIBUTING.md corto si mas adelante se abre algo del repo

    Estado actual:

    • labels y plantillas base ya existen en deskquest-docs
    • CHANGELOG.md y CONTRIBUTING.md ya existen en el repo principal
    • el canal publico de feedback ya opera desde deskquest-docs

    14. Revisar licencias y politica publica (completado)¶

    No hacia falta resolver aun la licencia comercial definitiva, pero si dejar claro:

    • que hoy no se concede licencia abierta sobre codigo, arte ni docs
    • que el portal publico existe para lectura, testing y soporte
    • que cualquier apertura futura requerira una decision explicita nueva

    Estado actual:

    • politica conservadora ya fijada en Politica publica de licencias
    • repo principal y portal publico no deben presentarse como open source en esta fase

    15. Hacer una pasada de privacidad (completado)¶

    Revisar antes de publicar fuera:

    • rutas locales personales en docs
    • nombres o datos que no convenga exponer
    • referencias internas innecesarias

    Este paso era importante porque el proyecto ya tenia bastante documentacion operativa detallada.

    Estado actual:

    • rutas locales personales reemplazadas por placeholders o rutas relativas
    • ejemplos de modding saneados para no arrastrar nombres propios innecesarios
    • worklog reciente corregido para no exponer rutas absolutas del autor
    • politica publica de privacidad fijada en Politica publica de privacidad

    16. Congelar la beta (completado)¶

    Una vez cerrados los pasos anteriores:

    • no abrir features grandes
    • no mezclar mejoras de producto con cierre de release
    • aceptar solo bugs bloqueantes y ajustes de entrega

    Estado actual:

    • la beta ya es operativa y publicable tecnicamente
    • la ronda principal de usabilidad, estetica y acceso ya esta cerrada
    • la build local 0.1.0-beta.2 ya se valido con una smoke test corta
    • la release publica nueva ya existe en deskquest-docs
    • el portal documental publico ya fue republicado con los enlaces correctos a 0.1.0-beta.2
    • la beta ya puede darse por compartible para colegas y testers cercanos
    • la referencia de polish ya queda fijada en Polish previo a compartir la beta

    17. Reanudar desarrollo funcional¶

    Solo despues de cerrar la beta operativa.

    Antes de abrir cualquiera de estas lineas, conviene cerrar una primera lectura de testers siguiendo la Operativa de feedback beta.

    Posibles siguientes lineas:

    • Cantera jugable
    • mejoras activas de Dirigir tala
    • integracion del lore
    • nuevas zonas o slices de game

    Prioridad inmediata recomendada¶

    Si hay que empezar ya, este es el orden corto:

    1. compartir la beta 0.1.0-beta.2
    2. recoger feedback real de colegas o testers cercanos
    3. decidir despues si la beta queda ya definitivamente congelada o si se abre el siguiente bloque funcional

    Lo que no conviene hacer aun¶

    • abrir nuevas features grandes
    • meter PixiJS ya
    • tocar la cantera jugable antes del cierre de beta
    • mezclar polish creativo con pasos de release y soporte

    Resultado esperado¶

    Si se sigue este orden:

    • DeskQuest tendra una primera beta publica limpia
    • la build compartible sera trazable y repetible
    • la documentacion publica no dependera del repo privado
    • el repo quedara presentable y operativo
    • y el proyecto podra volver al desarrollo funcional sin deudas de release
