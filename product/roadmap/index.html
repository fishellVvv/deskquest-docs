    Roadmap Tecnico¶

    0. Estado actual del proyecto¶

    La base tecnica inicial ya no esta en fase de arranque. A dia de hoy el proyecto ya tiene:

    • ventana idle funcional en escritorio
    • ventana dashboard separada
    • configuracion persistente
    • telemetria basica de pulsaciones
    • selector de dimension sincronizado entre ventanas
    • soporte real para tamano, monitor, esquina y always on top del idle
    • una Danger Zone funcional para reset protegido

    Situacion actual de la arquitectura de dimensiones¶

    El sistema ya no depende de dos dimensiones integradas compiladas. A dia de hoy:

    • builtin:medieval es la unica dimension integrada
    • el resto de dimensiones viven como dimensiones instaladas
    • los paquetes .dimension se importan manualmente y se descomprimen en la carpeta interna runtime
    • lunar-base.dimension actua como demo oficial de modding

    Eso significa que la migracion funcional principal ya esta cerrada y el frente actual ya no es documental, sino funcional: abrir el siguiente sistema critico del producto sin desviarnos de la fantasia desktop/idle/incremental.

    Prioridad absoluta actual¶

    La captura global ya ha quedado cerrada como base tecnica suficientemente estable. A partir de aqui, la prioridad vuelve a ser crecer en producto: dashboard, dominio compartido y primer slice real de game.

    Fase previa inmediata¶

    Antes de conectar el primer loop economico completo, se crea una capa de balance editable en tablas para evitar que los primeros costes y tiempos nazcan como constantes dispersas por el codigo.

    Regla para no desviarnos¶

    Mientras abrimos ese siguiente slice, evitaremos:

    • reabrir grandes refactors ya cerrados sin un motivo real de producto
    • convertir pequeños pulidos en foco principal si ya estan recogidos como pendientes diferidos
    • crecer UI sin acercarnos a economia, progreso o jugabilidad
    • perder de vista la fantasia desktop/idle/incremental que guia el proyecto

    1. Vision del proyecto¶

    Construir una app-juego de escritorio en la que "el mundo vive en el escritorio y a veces se abre a jugar".

    Documento de referencia de producto:

    • Vision DII

    El producto tendra tres niveles claramente separados pero conectados:

    1. idle: una estructura principal pequena en una esquina del escritorio, siempre visible, con feedback minimo.
    2. dashboard: una vista ampliada para consultar el estado del mundo, misiones, progreso, recursos y sistemas internos.
    3. game: una capa activa de juego lanzada desde edificios concretos e integrada dentro del mismo esquema mapa + pergamino.

    El objetivo tecnico principal es que el modo idle sea muy ligero y no cargue logica, assets o renderizados pesados cuando no sean necesarios.

    La primera ambientacion sera medieval y usara un castillo como skin principal, pero el nucleo del proyecto debe permitir sustituir esa ambientacion por otras sin rehacer la logica del juego.

    2. Decision tecnica recomendada¶

    Stack base¶

    • Tauri v2
    • Vite
    • React
    • TypeScript
    • React Context y almacenamiento local para la base actual
    • Zustand solo si el estado compartido crece lo suficiente como para justificarlo
    • PixiJS como renderer previsto para el nivel game

    Por que esta combinacion¶

    • Tauri permite construir una app de escritorio ligera, con ventanas nativas y buen control del comportamiento en escritorio.
    • React encaja muy bien para idle y dashboard, donde la UI es mas "app" que "juego".
    • PixiJS se puede cargar solo cuando haga falta y encaja mejor en actividades 2D integradas dentro del dashboard.
    • Todo el proyecto puede vivir en una sola app, con una sola base de codigo y un solo empaquetado.

    Lo que evitamos con esta decision¶

    • No dividir el proyecto entre "app por un lado" y "juego por otro".
    • No forzar un motor pesado para una experiencia que en su nucleo debe ser pequena y persistente.
    • No cargar el runtime del nivel game mientras el usuario solo esta viendo la estructura principal en una esquina.

    3. Principios tecnicos¶

    1. idle debe arrancar rapido.
    2. idle no debe cargar PixiJS ni ningun runtime del game.
    3. dashboard y game solo deben abrirse bajo demanda.
    4. El estado del mundo debe ser comun para los tres niveles.
    5. Los assets deben cargarse de forma diferida.
    6. La persistencia debe existir desde pronto, aunque al principio sea simple.
    7. Las metricas del sistema deben hacerse con cuidado para no convertir el proyecto en algo intrusivo o costoso.
    8. El dominio del juego no debe depender del skin activo.
    9. Textos, colores, tipografias y assets tematicos deben salir de una capa de tema o skin.

    3.1 Cambio critico en curso: migracion a dimensiones externas¶

    Este es el bloque de trabajo principal del proyecto hasta tener una version estable y firme del cambio.

    Objetivo del bloque¶

    Pasar del sistema actual de dimensiones compiladas a un sistema mixto y luego plenamente runtime con estas propiedades:

    • builtin:medieval permanece como dimension integrada por defecto y fallback universal
    • las dimensiones externas pueden cargarse desde carpeta o archivo .dimension
    • los textos core y los textos de skin se resuelven por capas distintas
    • el idle deja de depender de ramas de React hardcodeadas por dimension
    • los usuarios pueden compartir dimensiones sin recompilar la app

    Criterio de cierre del bloque¶

    Este cambio solo se considerara estable cuando se cumplan todos estos puntos:

    1. la app puede arrancar usando una dimension integrada o una externa
    2. si falta cualquier token, texto o asset, la resolucion cae limpiamente a builtin:medieval
    3. el idle usa assets y metadatos en vez de ramas por dimension
    4. el dashboard consume una dimension resuelta en runtime
    5. una dimension externa puede aportar idiomas propios
    6. el sistema sigue siendo seguro y no ejecuta codigo de terceros

    Secuencia cerrada de pasos¶

    Paso 1. Congelar el contrato del paquete de dimension¶

    Objetivo:

    • fijar el formato tecnico antes de seguir tocando runtime

    Estado:

    • completado

    Entregables:

    • docs/systems/dimensions/package-format.md
    • src/themes/shared/dimension-package.ts

    No seguimos al siguiente paso si:

    • el formato de paquete sigue siendo ambiguo
    • no estan claras las reglas de fallback, ids reservados o separacion entre core y skin

    Paso 2. Adaptador interno de dimensiones integradas¶

    Estado:

    • completado

    Objetivo:

    • hacer que medieval y lunar-base puedan verse como paquetes internos equivalentes, sin depender todavia de loader externo

    Tareas:

    • crear un adaptador que convierta las dimensiones integradas actuales a SkinPackageDefinition o ResolvedSkinPackageDefinition
    • dejar de asumir que el sistema de runtime trabaja solo con ThemeId cerrado
    • mantener compatibilidad con ThemeProvider mientras dure la migracion
    • fijar claramente builtin:medieval como dimension integrada canonical

    Criterio de salida:

    • las dos dimensiones actuales pueden representarse mediante el nuevo contrato runtime sin romper la app

    Paso 3. Separacion entre core i18n y skin i18n¶

    Estado:

    • completado

    Objetivo:

    • dejar de mezclar textos generales de la app con textos diegeticos de la dimension

    Tareas:

    • sacar los textos generales de configuracion, danger zone, errores y acciones comunes a un sistema core locales
    • mantener en la dimension solo los textos tematicos
    • conectar la preferencia idioma a una capa real de resolucion de textos
    • definir fallback de idiomas del core y fallback de idiomas de skin por separado

    Criterio de salida:

    • cambiar idioma afecta al core
    • cambiar dimension afecta a los textos tematicos
    • ambas capas pueden coexistir aunque no tengan exactamente los mismos idiomas

    Paso 4. Registro runtime de dimensiones¶

    Estado:

    • completado

    Objetivo:

    • sustituir el registro estatico de dimensiones por una abstraccion de biblioteca de dimensiones

    Tareas:

    • crear un registro runtime que mezcle dimensiones integradas y externas
    • resolver ids como string, no como union cerrada
    • adaptar almacenamiento y seleccion de dimension a ese modelo
    • garantizar que si una dimension seleccionada desaparece se vuelve limpiamente a builtin:medieval

    Criterio de salida:

    • el selector ya no depende de una lista compilada fija

    Paso 5. Refactor del idle a metadatos + assets¶

    Estado:

    • completado

    Objetivo:

    • eliminar el mayor bloqueo actual: el idle hardcodeado por rama de dimension

    Tareas:

    • sustituir IdleStructureArt.tsx basado en ramas por un renderer data-driven
    • leer layout.json
    • cargar idleStructure desde assets
    • calcular gateHitbox, gateFeedback, atmosphere y artworkLabel desde metadatos
    • conservar accesibilidad, escalado, monitor, esquina y posicionamiento ya resueltos

    Criterio de salida:

    • el idle puede renderizar una dimension nueva sin tocar JSX especifico de esa dimension

    Paso 6. Integracion del dashboard con dimensiones resueltas¶

    Estado:

    • completado

    Objetivo:

    • hacer que el panel consuma la misma resolucion runtime que el idle

    Tareas:

    • aplicar tokens y textos desde la dimension resuelta
    • revisar elementos del dashboard que todavia asumen solo medieval y lunar-base
    • dejar el selector de dimension preparado para bibliotecas dinamicas

    Criterio de salida:

    • dashboard e idle ya no dependen de dos sistemas distintos de dimension

    Paso 7. Loader de carpetas externas¶

    Estado:

    • completado

    Objetivo:

    • abrir la puerta al modding sin entrar todavia en empaquetado

    Tareas:

    • definir carpeta de usuario skins/
    • leer carpetas descomprimidas con manifest.json
    • validar esquema, rutas, locales y layout
    • mostrar en UI solo dimensiones validas
    • registrar errores de carga sin tirar la app abajo

    Criterio de salida:

    • un usuario puede copiar una carpeta de dimension y verla aparecer en la app

    Paso 8. Loader de archivos .dimension¶

    Estado:

    • completado

    Objetivo:

    • completar el formato de intercambio entre usuarios

    Tareas:

    • leer archivos .dimension como zip
    • validar contenido interno
    • resolver conflictos entre ids
    • definir politica de importacion, actualizacion y eliminacion

    Criterio de salida:

    • un usuario puede importar una dimension empaquetada y seleccionarla sin recompilar

    Paso 9. Endurecimiento y version estable¶

    Objetivo:

    • cerrar el cambio sin deuda tecnica critica

    Tareas:

    • pruebas manuales completas con fallback roto a proposito
    • pruebas de idiomas incompletos
    • pruebas de assets faltantes
    • pruebas de cambio en caliente entre integrada y externa
    • limpiar APIs temporales heredadas del sistema antiguo
    • documentar el flujo para creadores de dimensiones

    Criterio de salida:

    • podemos declarar estable el nuevo sistema de dimensiones y retomar crecimiento de dashboard y game

    4. Modelo de ventanas¶

    La app se planteara como una aplicacion Tauri con multiples ventanas.

    Ventana 1: idle¶

    • Tamano pequeno
    • Sin bordes
    • Siempre visible
    • Posicionada en una esquina
    • Transparente o casi transparente
    • Muestra solo la estructura principal del skin activo y pequenas senales visuales
    • Permite click en el punto interactivo principal para abrir el dashboard

    Ventana 2: dashboard¶

    • Se abre bajo demanda
    • Tamano medio
    • UI de gestion del mundo o asentamiento activo
    • Presenta datos, misiones, inventario, progreso, logs y acciones simples
    • Puede cerrarse o minimizarse sin afectar al idle

    Capa 3: game¶

    • Se activa bajo demanda desde dashboard
    • Reutiliza la zona izquierda del reino para mostrar la escena jugable
    • Mantiene el pergamino derecho como panel de mision o control
    • Debe destruirse o suspenderse cuando no se use

    Regla clave¶

    Al arrancar la app solo debe existir la ventana idle. dashboard no debe inicializarse si el usuario no lo ha abierto, y el runtime de game no debe cargarse mientras no se lance una actividad manual.

    5. Arquitectura general¶

    5.1 Capas¶

    Capa de escritorio¶

    Responsable de:

    • crear y posicionar ventanas
    • gestionar ciclo de vida
    • persistir datos locales
    • emitir eventos entre ventanas

    Esta capa vivira en src-tauri/.

    Capa de interfaz¶

    Responsable de:

    • renderizar idle
    • renderizar dashboard
    • montar la experiencia game

    Esta capa vivira en src/.

    Capa de dominio¶

    Responsable de:

    • estado del mundo
    • recursos
    • progreso
    • misiones
    • eventos del usuario
    • reglas de desbloqueo

    Esta capa debe ser compartida por los tres niveles.

    5.2 Estado compartido¶

    El proyecto necesita una fuente de verdad clara.

    Recomendacion¶

    • Estado ligero en frontend con Context mientras el proyecto siga pequeno
    • Persistencia local simple en esta fase
    • Sincronizacion entre ventanas mediante almacenamiento compartido, eventos y capacidades Tauri

    Idea practica¶

    1. Al arrancar idle, WorldProvider carga el estado persistido.
    2. Las escrituras del mundo pasan por helpers de dominio y almacenamiento comun.
    3. Cuando dashboard o game modifican algo importante, se persiste y se emite un evento.
    4. idle escucha esos eventos y actualiza su feedback visual.

    Esto evita depender de hacks entre ventanas y deja una base robusta para crecer.

    5.3 Carga diferida¶

    Para que idle sea ligero:

    • dashboard y game deben cargarse de forma lazy
    • PixiJS debe importarse dinamicamente cuando una actividad lo necesite
    • los assets grandes no deben precargarse en el arranque
    • las animaciones de la estructura principal deben ser simples y baratas

    Regla¶

    Nada del nivel game debe entrar en el bundle inicial del modo idle salvo interfaces minimas de comunicacion.

    7. Decisiones tecnicas importantes desde el principio¶

    7.1 Sistema de skins y ambientaciones¶

    La ambientacion no debe vivir mezclada con la logica del juego.

    Regla base¶

    La dimension medieval del castillo es solo la primera presentacion del sistema y tambien el fallback universal del proyecto.

    Modelo objetivo¶

    • builtin:medieval vive dentro de la app y nunca desaparece
    • las dimensiones externas viven fuera del bundle
    • una dimension externa puede distribuirse como carpeta o .dimension
    • toda dimension externa se resuelve contra builtin:medieval

    Cada dimension debe agrupar¶

    • assets visuales
    • tokens de color
    • tipografias
    • nombres y textos tematicos
    • iconografia
    • pequenas variantes de sonido si en el futuro existen

    Cada dimension no debe cambiar¶

    • reglas del juego
    • estructura del estado
    • ids internos
    • contratos de eventos

    Contrato minimo de una dimension¶

    • schemaVersion
    • id
    • label
    • description
    • version
    • defaultLocale
    • tokens
    • locales
    • idleLayout
    • assets
    • uiHints opcional

    Decision ya tomada¶

    La segunda dimension placeholder ya sirvio para validar el desacoplamiento inicial. A partir de aqui, el siguiente paso correcto no es anadir mas dimensiones integradas, sino migrar el sistema a paquetes externos siguiendo ../systems/dimensions/package-format.md.

    7.2 Animacion de la estructura principal¶

    No usar un GIF como solucion final salvo para una prueba inicial.

    Mejor enfoque¶

    • sprite sheet
    • webp animado
    • pequenas capas HTML/CSS
    • o un canvas muy simple si hace falta

    Motivo¶

    Necesitamos poder encender pequenas senales de estado sin regenerar un archivo completo cada vez.

    7.3 Vocabulario desacoplado del dominio¶

    La logica interna no debe usar nombres medievales como base del modelo.

    Recomendacion¶

    • usar ids y nombres neutrales en el dominio
    • resolver el texto visible mediante lexicon del skin activo
    • evitar hardcodear palabras como castillo, reino o taberna en componentes compartidos

    Ejemplo de enfoque¶

    • dominio: WorldState, MissionState, ResourceState, PrimaryStructure
    • skin medieval: WorldState se presenta como Reino
    • skin espacial: WorldState se presenta como Base

    Regla¶

    Si un concepto solo existe por ambientacion, debe vivir en la skin y no en el nucleo.

    7.4 Registro de interacciones¶

    Separar dos casos:

    Caso A: interacciones dentro de la app¶

    • tiempo de ejecucion
    • clicks dentro de las ventanas
    • teclas pulsadas mientras la app tiene foco

    Esto se puede implementar pronto y con poco riesgo.

    Caso B: interacciones globales del sistema¶

    • clicks fuera de la app
    • teclas del sistema entero

    Esto debe dejarse para mas adelante y solo si de verdad aporta valor, porque complica permisos, privacidad y mantenimiento.

    Decision actualizada¶

    El proyecto ya ha abierto un primer corte del caso B, pero con una regla muy estricta: solo se agregan pulsaciones totales y no se guarda contenido, teclas concretas ni secuencias del usuario.

    7.5 Persistencia¶

    Empezar simple.

    Fase inicial¶

    • guardar estado en almacenamiento local sencillo
    • snapshots del mundo
    • configuracion de ventanas

    Fase posterior¶

    • migrar a SQLite si aparecen sistemas complejos
    • historico de misiones
    • inventario grande
    • estadisticas acumuladas

    Regla¶

    No introducir base de datos relacional el primer dia si aun no sabemos el volumen real del dominio.

    7.6 Navegacion entre niveles¶

    La navegacion no debe depender solo de rutas web. Debe existir un modelo explicito de ventanas.

    Flujo esperado¶

    1. La app arranca en idle.
    2. Click en el punto de entrada principal del skin.
    3. Se abre dashboard.
    4. Desde dashboard, el usuario puede lanzar game.
    5. Al cerrar game, se conserva el estado del mundo y idle refleja cambios.

    Fase 0 - Preparacion del proyecto¶

    • Definir contrato tecnico de skin
    • Crear tema visual base
    • Crear una skin medieval inicial
    • Crear una segunda skin placeholder para validar desacoplamiento

    Entregable¶

    Proyecto arrancando en local con una ventana Tauri simple.

    Fase 1 - Vertical slice del modo idle¶

    Objetivo¶

    Conseguir la primera estructura funcional en una esquina del escritorio.

    Tareas¶

    • Crear ventana idle
    • Configurar tamano, transparencia, posicion y siempre visible

    • Definir zona clicable del acceso principal
    • Crear micro feedback visual
    • Registrar tiempo de ejecucion local

    • sin polling agresivo
    • sin assets pesados
    • sin loops complejos

    Entregable¶

    Una app que vive en una esquina, reacciona minimamente y abre el siguiente nivel con la primera skin medieval ya montada.

    Fase 1.5 - Direccion visual del idle y dashboard¶

    Objetivo¶

    Convertir el prototipo tecnico actual en una base visual alineada con la fantasia del proyecto antes de seguir anadiendo mas sistemas.

    Contexto¶

    El primer cambio idle -> dashboard ya esta validado tecnicamente, pero su presentacion actual solo debe entenderse como una prueba de flujo y no como la solucion visual final.

    Tareas¶

    • definir la metafora visual de apertura del dashboard
    • decidir si el dashboard emerge desde la puerta, se despliega sobre la estructura o se abre como pieza independiente
    • fijar una composicion base del idle menos "tarjeta" y mas "objeto vivo en escritorio"
    • fijar como objetivo del idle una huella aproximada de 320x160 px, anclada en la esquina inferior derecha
    • priorizar que el idle muestre solo la estructura principal y senales minimas, sin UI de panel clasica
    • orientar la primera skin medieval hacia un castillo pixelart sencillo, legible en tamano pequeno y mirando hacia la izquierda
    • separar mejor arte de escena, zonas clicables y capas de interfaz
    • crear una version wireframe o mock funcional del idle
    • crear una version wireframe o mock funcional del dashboard
    • validar que el lenguaje visual elegido funcione tambien para otras skins

    Regla¶

    No seguir creciendo en contenido del dashboard hasta que exista una direccion visual clara para el binomio idle/dashboard.

    Entregable¶

    Una maqueta funcional que ya se parezca a la experiencia deseada, aunque todavia use datos placeholder.

    Referencia actual de diseno¶

    • idle: castillo pequeno apoyado abajo a la derecha, con un ancho cercano al reloj de Windows
    • tamano orientativo: 320x160
    • estilo buscado: pixelart sencillo, legible y con silueta clara
    • la ambientacion medieval debe priorizar lectura de volumen, almenas, torres y puerta antes que detalle fino

    El usuario percibe que todo forma parte del mismo mundo.

    10. Riesgos tecnicos y mitigacion¶

    Riesgo 1: el nivel game contamina el arranque¶

    Mitigacion¶

    • import dinamico
    • integracion embebida en dashboard
    • bundle separado

    Riesgo 2: el estado se duplica entre ventanas¶

    Mitigacion¶

    • fuente de verdad clara
    • comandos backend
    • eventos tipados

    Riesgo 3: demasiada complejidad demasiado pronto¶

    Mitigacion¶

    • empezar con placeholders
    • priorizar vertical slices
    • evitar sistemas "definitivos" en la primera iteracion

    Riesgo 4: la estructura principal deja de ser ligera¶

    Mitigacion¶

    • medir
    • no ejecutar logica innecesaria
    • no cargar librerias de juego en idle

    Riesgo 5: acoplar la logica a la ambientacion medieval¶

    Mitigacion¶

    • usar nombres internos neutrales
    • crear contrato de skin desde el inicio
    • probar pronto una segunda skin placeholder
    • no hardcodear textos tematicos en componentes compartidos

    11. Criterios de exito del MVP¶

    El MVP debe considerarse valido cuando cumpla esto:

    1. La app arranca solo con idle.
    2. La estructura principal aparece en una esquina y responde visualmente.
    3. El punto de entrada principal abre dashboard.
    4. dashboard muestra datos reales del mundo, aunque sean pocos.
    5. Existe guardado y carga del progreso.
    6. Se puede abrir un primer game muy simple.
    7. Lo que ocurre en game impacta en dashboard y idle.
    8. El consumo del modo idle se mantiene bajo.
    9. Existe al menos una segunda skin placeholder que cambia nombres, colores y assets sin tocar la logica central.

    14. Direccion confirmada para dimensiones¶

    Se fija esta direccion para no desviar la migracion:

    1. builtin:medieval sera la unica dimension integrada y el fallback absoluto del sistema.
    2. lunar-base dejara de tratarse como built-in permanente y pasara a convertirse en una dimension opcional distribuida como .dimension.
    3. El resto de dimensiones opcionales se distribuiran tambien como un unico archivo .dimension.
    4. Las dimensiones opcionales cargadas por la app viviran descomprimidas en una carpeta interna de dimensiones instaladas.
    5. Los archivos .dimension funcionaran como paquetes importables y compartibles, no como carpeta runtime escaneada permanentemente.
    6. lunar-base.dimension funcionara como demo oficial de modding y debera incluir documentacion suficiente para que un usuario pueda usarla como referencia.

    Implicaciones practicas¶

    • no debemos seguir anadiendo nuevas dimensiones opcionales como built-in
    • el runtime debe pensar en dimensiones instaladas, no en carpetas arbitrarias que se escanean continuamente
    • importar una dimension sera una accion explicita del usuario
    • cargar las dimensiones ya instaladas sera automatico al arrancar
    • lunar-base.dimension se tratara como demo oficial instalable, no como dimension preactivada
