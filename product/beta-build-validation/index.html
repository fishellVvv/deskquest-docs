    Validacion de build beta¶

    Esta pagina fija la comprobacion minima que conviene hacer sobre una build empaquetada antes de compartirla.

    1. Objetivo¶

    Confirmar que el comportamiento estable en dev sigue siendo real cuando la app se ejecuta como build de escritorio.

    2. Comando de build¶

    La build de escritorio debe prepararse con:

    npm run tauri -- build

    Con la configuracion actual de Tauri:

    • el frontend se compila antes con npm run build
    • la app se empaqueta bajo src-tauri/target/release/bundle/

    Para la build candidata actual, los artefactos esperados son:

    • DeskQuest_0.1.0-beta.2_x64_en-US.msi
    • DeskQuest_0.1.0-beta.2_x64-setup.exe

    Release publica viva:

    • https://github.com/fishellVvv/deskquest-docs/releases/tag/v0.1.0-beta.2

    Nota para Windows:

    • si la version publica usa prereleases tipo 0.1.0-beta.2, el target msi necesita una version numerica paralela en bundle.windows.wix.version porque WiX no acepta identificadores prerelease textuales en ProductVersion

    3. Preparacion recomendada¶

    Antes de validar la build:

    1. cerrar la app si estaba abierta
    2. dejar el progreso normal activo
    npm run dev:preset:none
    1. decidir si la prueba se hace con perfil limpio o con progreso previo

    4. Checklist minima sobre build empaquetada¶

    Arranque¶

    1. instalar o lanzar la build
    2. comprobar que el idle aparece en escritorio
    3. comprobar que existe icono de bandeja

    Reino¶

    1. abrir dashboard
    2. cerrar y reabrir varias veces
    3. comprobar que no se duplica la ventana
    4. comprobar que la shell mapa + pergamino se ve bien

    Persistencia¶

    1. generar progreso real
    2. cerrar la app por completo
    3. volver a abrir la build
    4. confirmar que el reino sigue coherente

    Slice activo¶

    1. entrar en Dirigir tala
    2. jugar una mision corta
    3. volver al reino
    4. comprobar que la economia entregada al mundo coincide con la expedicion

    5. Criterio de salida¶

    La build se considera lista para testers si:

    • arranca sin ajustes manuales raros
    • el idle es visible
    • el dashboard no se duplica
    • la persistencia funciona
    • Dirigir tala no rompe el reino

    6. Si algo falla solo en build¶

    Prioridad de revision:

    1. rutas de assets
    2. visibilidad del idle
    3. apertura del dashboard
    4. bandeja del sistema
    5. persistencia y reentrada

    Si un bug solo existe en build, debe tratarse como mas grave que un bug de dev, porque impacta directamente a testers.
