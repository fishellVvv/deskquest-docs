    Politica de versionado¶

    Estado actual¶

    El sistema profesional de versionado ya esta activo.

    Hoy conviven tres capas distintas y separadas:

    1. una version publica del producto
    2. un hito interno ligado al worklog
    3. una version del guardado

    Valores activos:

    • version publica del producto: 0.1.0-beta.2
    • release publica viva: 0.1.0-beta.2
    • hito interno actual: 0.0.255a
    • save schema version: 1

    1. Version publica del producto¶

    DeskQuest usa ya SemVer con prereleases como linea publica.

    Formato¶

    MAJOR.MINOR.PATCH[-prerelease]

    Ejemplos:

    • 0.1.0-beta.1
    • 0.1.0-beta.2
    • 0.1.0-rc.1
    • 0.1.0

    Regla de incremento¶

    PATCH¶

    Para:

    • bugfixes
    • QA fixes
    • ajustes de UX
    • balance
    • correcciones documentales

    Siempre que no cambien de forma fuerte el alcance del producto.

    MINOR¶

    Para:

    • un nuevo slice jugable importante
    • una nueva zona o capa relevante
    • un sistema nuevo visible para el jugador
    • un salto real de alcance del producto

    MAJOR¶

    Para:

    • cambios incompatibles fuertes
    • cambios deliberados que rompan saves o modding sin compatibilidad
    • reposicionamiento importante del producto

    2. Hito interno de trabajo¶

    La trazabilidad interna sigue ligada al WORKLOG.

    Formato¶

    0.0.<entrada><letra>

    Ejemplos:

    • 0.0.168a
    • 0.0.169a
    • 0.0.169b

    Regla de lectura¶

    • 0.0 indica fase pre-1.0
    • <entrada> coincide con la ultima entrada relevante del WORKLOG
    • <letra> identifica subiteraciones internas dentro del mismo hito

    Regla practica¶

    1. nueva entrada relevante del WORKLOG
    2. nueva base numerica
    3. reinicio de letra a a

    Este valor debe entenderse como:

    • trazabilidad interna
    • lectura rapida del estado de trabajo
    • referencia para STATUS, versioning y worklog

    No sustituye a la version publica del producto.

    3. Version del guardado¶

    El proyecto tiene ya una version explicita de persistencia.

    Valor activo:

    • save schema version: 1

    Regla:

    • solo sube si cambia el formato del guardado
    • no sube por balance, docs o polish visual
    • debe permitir migraciones claras entre perfiles viejos y nuevos

    Estado actual del runtime¶

    El mundo se persiste ya como un sobre con schema:

    • saveSchemaVersion
    • world

    Compatibilidad actual:

    • si existe un guardado legacy plano, el loader lo sigue aceptando
    • al persistir de nuevo, queda ya reescrito con save schema version: 1

    Politica de tags y releases¶

    La release publica actual debe reflejarse asi:

    • tag recomendado: v0.1.0-beta.2
    • release recomendada: DeskQuest 0.1.0-beta.2

    Ese numero debe reflejarse de forma consistente en:

    • package.json
    • src-tauri/Cargo.toml
    • src-tauri/tauri.conf.json
    • notas de release
    • build compartida a testers

    Nota especifica para MSI en Windows¶

    El bundle msi de Tauri no puede derivar directamente una version prerelease con identificador textual como 0.1.0-beta.1.

    Por eso, cuando la version publica use prereleases, WiX debe llevar una version numerica paralela en:

    • src-tauri/tauri.conf.json
    • bundle.windows.wix.version

    Para la linea actual:

    • version publica: 0.1.0-beta.2
    • version msi: 0.1.0.2

    Regla recomendada:

    • 0.1.0-beta.1 -> 0.1.0.1
    • 0.1.0-beta.2 -> 0.1.0.2
    • 0.1.0-rc.1 -> 0.1.0.101 o equivalente documentado

    Lo importante es mantener una secuencia numerica valida para Windows MSI sin perder la version publica legible del producto.

    La transicion oficial ya ha quedado cerrada asi:

    1. se mantuvo 0.0.x como trazabilidad interna
    2. se completo el rename tecnico a DeskQuest
    3. se activo la primera linea publica real
    4. se fijo la persistencia en save schema version: 1

    Siguiente paso operativo¶

    La release 0.1.0-beta.2 sigue siendo la referencia publica viva, pero el bloque inmediato del repo ya no es de distribucion ni de vision de producto, sino de ejecucion del primer slice de midgame: los dos primeros cortes reales del Bloque A ya meten Puesto de caza + comida + bienestar, y el remate corto actual ya corrige la pausa real de crecimiento bajo bienestar fragil, refuerza la lectura UX del patio sobre Viviendas y Puesto de caza, y deja ya cerrado Bloque A en docs fuente y portal publico antes del paso posterior a Taberna + tablon.

    1. usar ADR-0001 y ADR-0002 como marco normativo
    2. dar por cerrado M0, ya materializado con reglas, smoke test y baseline visibles
    3. dar por cerrado M1, ya validado con smoke test final tras separar world-model, world-selectors, world-commands y world-simulation
    4. dar por cerrado M2, ya validado con build limpia, runtime explicitado, backend transicional separado, primer precedente correcto de game y smoke test final pasado
    5. dar por cerrado M3 con foco en las grandes superficies del dashboard, empezando por descargar DashboardSettingsRoom por integraciones antes de tocar el patio grande; ese primer corte ya existe con hooks propios para monitores, documentacion, dimensiones y Danger Zone, y el segundo corte ya saca tambien la seleccion local del patio a useCourtyardSelection; el tercer corte ya saca tambien el wiring de Dirigir tala a useCourtyardLoggingMission; el cuarto corte ya saca el overview a CourtyardOverviewLedger; el quinto ya saca tambien el ledger del edificio principal a CourtyardPrimaryStructureLedger; el sexto ya saca tambien el ledger del edificio seleccionado a CourtyardSelectedBuildingLedger; el septimo ya saca tambien la hoja de nombrado inicial a CourtyardSettlementNamingLedger; el octavo ya saca tambien el overlay de QA a CourtyardTestingToolsOverlay; el noveno deja ademas de enrutar las lecturas del dashboard por world-state; el smoke test final ya ha pasado y M3 queda cerrado. El siguiente paso real pasa a ser decidir si se abre M4 directamente o si conviene antes una preparacion tecnica corta; esa M4 ya queda abierto con siete cortes reales: el primero abre platform/world, el segundo deja world-state como shim puro, el tercero retira ya esa fachada del codigo vivo y el cuarto estrecha world-storage para dejarlo solo con acciones persistidas del mundo; el quinto mueve esas acciones a application/world y retira ya world-storage del codigo vivo; el sexto empieza ya el cleanup de legacy pre-beta.2 fuera de world en configuration y telemetry; el septimo remata ese cleanup corto en themes/shared; el smoke test final ya ha pasado y M4 queda cerrado
    6. abrir ahora un M4b corto de cleanup residual de legacy, separado del milestone principal, para revisar la migracion nativa del perfil testgame, el puente legacy de settlementName, browser-storage-migration.ts y el mapa legacy de ids de dimension; el primer corte ya retira la migracion nativa del perfil testgame y el segundo retira el puente legacy de settlementName; el tercero reduce browser-storage-migration.ts a su unica utilidad compartida con consumidores vivos; el cuarto retira ya el mapa legacy de ids de dimension en runtime, el smoke test final pasa limpio y M4b queda cerrado
    7. abrir ahora M5 con una mision concreta: consolidar la arquitectura viva y preparar el primer slice post-refactor del midgame, evitando que el milestone se convierta en una bolsa abstracta de "mas documentacion"
    8. completar M5-1 creando una pagina corta de arquitectura actual, de modo que la forma viva del proyecto deje de depender del historial del refactor
    9. completar M5-2 fijando reglas de entrada de nuevos sistemas, de modo que el primer slice de midgame no vuelva a mezclar dominio, aplicacion, plataforma, UI y dimensiones desde su nacimiento
    10. completar M5-3 bajando ya a la game bible el primer arco jugable del midgame, dejando bienestar, delegaciones, Taberna, comida y territorio local como direccion operativa antes de definir el blueprint del slice
    11. completar M5-4 creando un blueprint operativo del primer slice post-refactor con alcance, no-alcance y dos bloques recomendados de implementacion: Puesto de caza + comida + bienestar y Taberna + tablon + primeras misiones
    12. completar M5-5 cerrando el milestone con el paquete ya consolidado de arquitectura viva, reglas de entrada, canon del primer midgame y blueprint del slice inicial, dejando el siguiente paso ya en producto
