    Estrategia de distribucion beta¶

    Estado¶

    • aceptada
    • fecha: 2026-03-24

    Decision¶

    Para la beta de DeskQuest, el canal oficial de distribucion sera:

    • GitHub Releases

    Y el proyecto no usara por ahora:

    • GitHub Packages

    Por que GitHub Releases¶

    La beta necesita distribuir:

    • instaladores descargables
    • versiones concretas del juego
    • notas de release
    • enlaces claros para testers

    GitHub Releases encaja bien porque permite:

    • crear un tag versionado, por ejemplo v0.1.0-beta.1
    • asociar una pagina publica a esa version
    • adjuntar binarios:
    • .msi
    • setup.exe
    • dejar notas de release reutilizables para QA y testers

    Para esta fase, Releases funciona como:

    • historial de builds compartibles
    • punto oficial de descarga
    • referencia publica de cada beta

    Por que no GitHub Packages¶

    GitHub Packages esta pensado para distribuir:

    • paquetes npm
    • contenedores
    • artefactos consumibles por otros proyectos
    • dependencias tecnicas para desarrolladores

    Ahora mismo DeskQuest no necesita nada de eso.

    La unidad de distribucion actual es:

    • una app de escritorio jugable

    no:

    • una libreria
    • un SDK
    • un paquete para instalar desde un gestor de dependencias

    Por eso GitHub Packages no aporta valor claro en esta fase y solo anadiria ruido operativo.

    Regla operativa¶

    Mientras la beta siga en este formato:

    • cada build compartible debe publicarse como GitHub Release
    • los artefactos oficiales deben adjuntarse en la release
    • GitHub Packages no debe configurarse ni mantenerse

    Formato recomendado de release¶

    Tag¶

    • v0.1.0-beta.1

    Titulo¶

    • DeskQuest 0.1.0-beta.1

    Assets¶

    • DeskQuest_0.1.0-beta.1_x64-setup.exe
    • DeskQuest_0.1.0-beta.1_x64_en-US.msi

    Estado¶

    • marcar la release como Pre-release

    Regla para testers¶

    Si se comparte una beta:

    • el enlace principal debe ser la pagina de GitHub Release
    • el binario recomendado puede ser el setup.exe
    • el .msi queda como alternativa
    • la documentacion de apoyo debe ir fuera del binario

    Cuando revisar esta decision¶

    Solo conviene reabrir esta politica si ocurre al menos uno de estos casos:

    • aparece un launcher propio
    • se automatiza un pipeline de releases mas complejo
    • el proyecto empieza a publicar tooling o paquetes tecnicos reutilizables
    • cambia el modelo de distribucion de la beta o de la release publica
